# 機能要望整理

あなたはCiviLinkのカスタマーサクセス担当として、顧客からの機能要望を構造化し、プロダクトチームへのフィードバックを準備します。

## 引数
$ARGUMENTS に顧客名と要望内容が含まれています。

## 実行手順

### Step 1: コンテキスト収集
以下を並列で確認:
- `customers/{顧客名}/README.md` で顧客の利用状況・契約状況を確認
- `product/features/` で既存のフィーチャーリクエストを検索（重複確認）
- `product/known-issues/` で関連する既知課題がないか確認
- `customers/` 全体で同じ要望を持つ他の顧客を検索

### Step 2: 要望の構造化

```
## 機能要望: {タイトル}
- 顧客: {顧客名}
- 担当者: {担当者名}
- 日付: {日付}
- カテゴリ: {UI/UX / 図面表示 / 照査機能 / 連携 / パフォーマンス / その他}
- 優先度: {P0(ブロッカー) / P1(重要) / P2(あると嬉しい) / P3(将来的に)}

### 要望内容
{顧客の言葉そのまま}

### 背景・理由
{なぜその機能が必要か、業務上の課題}

### 技術的評価
- 既存機能で対応可能か: {はい/いいえ — 可能な場合はその方法}
- 技術的制約: {例: DocuWorks SDK制約}
- 代替アプローチ: {別の解決方法がある場合}

### 影響範囲
- 同じ要望を持つ顧客: {顧客名リスト}
- 有償化への影響: {この要望が有償化判断に影響するか}
- ロードマップとの整合: {既存の計画との関係}

### 推奨アクション
- {プロダクトチームへの提案}
```

### Step 3: 記録
- `customers/{顧客名}/issues.md` に要望を記録
- 重要度が高い場合は `tasks/todo.md` にフォローアップタスクを追加

## 重要なルール
- 顧客の言葉を勝手に解釈しない。原文を保持した上で構造化する
- DocuWorks関連の要望は技術的制約を必ず記載
- .envファイルは絶対に読まない
