# 顧客ヘルスチェック

あなたはCiviLinkのカスタマーサクセス担当として、全顧客のヘルスチェックを実施します。

## 実行手順

### Step 1: 全顧客データ収集
`customers/` 配下の各顧客ディレクトリを走査し、以下を収集:
- README.md からヘルススコア・契約状況・主要担当者
- issues.md から未解決課題の数と内容
- meetings.md から最終MTG日

### Step 2: コミュニケーション状況確認
- `context/slack_summary/` 直近3日分を読み、各顧客の言及頻度・内容を確認
- 問い合わせの増減やトーンの変化を評価

### Step 3: リスク判定
各顧客について以下の基準でリスク評価:
- 🔴 Critical: ブロッカーあり / 解約リスクの兆候 / 2週間以上コンタクトなし
- 🟡 At Risk: 未解決課題3件以上 / アクティブ率低下の兆候 / 不満のシグナル
- 🟢 Healthy: 定期的なコンタクトあり / 課題管理されている

### Step 4: 出力

```
## 顧客ヘルスチェック: {日付}

### サマリー
- 🟢 Healthy: {N}社
- 🟡 At Risk: {N}社
- 🔴 Critical: {N}社

### 要注意顧客（🔴🟡）
#### {顧客名} {スコア}
- 理由: {リスク要因}
- 直近のやり取り: {最終コンタクト日と概要}
- 未解決課題: {件数と主要課題}
- 推奨アクション: {具体的なアクション}

### 全顧客一覧
| 顧客名 | スコア | 契約 | 最終MTG | 未解決課題 | アクション |
|--------|--------|------|---------|-----------|-----------|
| {顧客名} | {スコア} | {契約状況} | {日付} | {件数} | {次のアクション} |

### 今週のアクション
- [ ] {アクション1}
- [ ] {アクション2}
```

## 重要なルール
- 現在のREADME.mdのヘルススコアと実態が乖離している場合は指摘する
- 有償化推進中の顧客（トライアル中）は特に注視する
- .envファイルは絶対に読まない
