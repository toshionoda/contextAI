# Slack → Mazrica 自動登録の仕組み提案

**作成日**: 2026-04-14
**作成者**: 野田
**共有先**: 川端部長
**目的**: CRMインプットの形骸化を解消する自動化方針の協議

---

## 背景・課題

- Mazricaの**名刺OCRは機能している**が、商談ステータス更新は形骸化（UIが重い・入力項目が多い）
- 営業チーム追加に伴い、情報の分断・属人化リスクが拡大
- 現状: 名刺OCR以外の情報は結局Excel/口頭で管理されている

## 提案: 「Slackに雑に投げるだけ」で Mazrica に自動登録

### 営業メンバーがやること（これだけ）

Slack `#crm-inbox` チャンネルに以下を投げるだけ:

| 投げるもの | 例 |
|---|---|
| 名刺写真 | スマホで撮ってSlackに貼る |
| 音声メモ | 商談帰りにSlackで音声録音「○○建設の田中部長、CiviLink興味あり。来週見積もり送る」 |
| テキストメモ | 「JRC小林さんと電話。DocuWorks検証、来月結論出す」 |

### 裏側で起きること（自動）

1. **名刺写真** → AI（Gemini Vision）が会社名・氏名・連絡先を読み取り
2. **音声メモ** → AIが文字起こし → 会社・担当者・商談内容・ネクストアクションを構造化
3. **テキスト** → AIが同様に構造化
4. **Mazricaに自動登録**:
   - 取引先が既存 → 情報を更新
   - 取引先が新規 → 新規登録 + コンタクト作成
   - 案件情報あり → 案件を作成 or 更新
5. **Slack返信で確認**: 元の投稿のスレッドに登録結果を返信

```
✓ 新規リード登録
株式会社ABC 田中太郎様（設計部長）
プロダクト: CiviLink
金額: 500万円（確度50%）
Next: 見積もり送付（2026-04-21）
```

## 既存レコードの更新にも対応

会社名だけでなく、**部署 + 氏名**の組み合わせで既存コンタクトを自動推定:

| Slackへの投稿 | 推定ロジック |
|---|---|
| 「JRC小林さんに電話した」 | 会社名 + 氏名 → 取引先JRC内の小林さんを特定 |
| 「設計部の田中さんと打合せ」 | 部署 + 氏名 → コンタクト検索で逆引き |
| 「田中部長、来月見積もり出す」 | 氏名のみ → 候補が1件なら自動採用 |

- 候補が複数ヒットした場合はスコアリング（会社名一致+2、部署一致+1）で最有力を選択
- 特定できない場合はSlackで「候補が複数あります」と通知（誤登録を防止）

## Mazrica側のメリット

- **名刺OCR以外の情報もMazricaに入る** → 一元管理が実現
- 営業はMazricaのUIを直接触る必要がない
- 新規登録だけでなく**既存レコードの更新もSlackから可能**
- 既存データとの重複チェック付き（二重登録を防止）
- Mazrica Open API（POST/PATCH）を利用するため、正規の方法で連携

## 技術面

| 項目 | 内容 |
|---|---|
| 実装状況 | プロトタイプ完成済み（スクリプト2本） |
| 利用API | Mazrica Open API / Gemini Vision & Audio / Claude API |
| 実行方法 | 始業コマンド（start-work.sh）に組み込み済み。毎朝自動処理 |
| 追加コスト | なし（既存のAPI契約内で対応可能） |
| 必要な準備 | Slack `#crm-inbox` チャンネル作成、Mazrica APIキー設定 |

## 協議事項

1. **Slackチャンネル運用**: `#crm-inbox` を営業チーム全員に周知してよいか
2. **Mazrica案件タイプ**: 自動登録時のデフォルト案件タイプIDをどれにするか
3. **段階的導入**: まず野田が1週間テスト運用 → 問題なければ営業チームに展開、でよいか
4. **将来的なMazrica廃止検討**: この仕組みが定着すれば、Mazrica → 別ツール（Notion等）への移行も容易（出力先を差し替えるだけ）

## 期待効果

- 営業の**CRMインプット工数**: 1件あたり5分 → ほぼ0分（Slackに投げるだけ）
- **情報鮮度**: 商談直後にSlackに投げる運用が定着すれば、常に最新
- **引き継ぎ**: Mazricaに全情報が集約されるため、属人化を解消
