# 検証した内容
精度を左右する4つの主要パラメータを個別に検証しましたが、いずれもトレードオフが構造的に存在し、単独調整での改善は頭打ちでした。

1. **赤色領域の検出感度（HSV 閾値）**
   厳しくすると CAD 印字数字の誤検出は減るが、薄い赤字・小さい数字 158 件を取りこぼし、F1 はむしろ -0.003 悪化。

2. **朱書きクラスタの統合距離**
   100px → 50px に半減しても TP は +4 件（全 794 件中・+0.7%）のみ。

3. **最小クラスタ面積**
   下げるとノイズを拾い、上げると小数点付き寸法（例: 147.7）を取り逃す。両立不能。

4. **Gemini の OCR 判定揺れ**
   temperature=0 でも、正解 411 件中 101 件（25%）が実行ごとに揺れます。N=3 多数決サンプリングも試しましたが F1 は変わらず、API コストが 3 倍になるだけという結果でした。

# なぜ「構造的な限界」なのか
現アーキテクチャは「OpenCV で赤領域を絞る → Gemini で文字化する」の 2 段構成です。
・ ①の段階で「朱書き手書き」と「CAD 印字の赤い数字」「細い赤線ノイズ」が視覚情報だけでは分離しきれない
・ ②の段階で Gemini の判定が確率的に揺れる

この 2 点が F1 の上限を作っており、個別パラメータの調整では超えられません。
