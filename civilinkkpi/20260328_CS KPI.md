# CSチーム KPI設計（R8）

> 本資料は、来年度のCSチームKPIについて、野田・内田・川畑・武井の4名で目線合わせを行うための叩き台です。
> 設計の意図や経緯を共有し、議論を通じて最終決定に持っていくことを目的としています。

## 1. エグゼクティブサマリー

### チーム共通KPI

| KPI | 指標 | 目標値 |
|---|---|---|
| スナップショットARR | 3月末時点の有償ユーザー数 × ユーザー単価 × 12 | 要設定 |

### 個人KPI一覧

| メンバー | 担当領域 | KPI | 目標値 | 評価サイクル |
|---|---|---|---|---|
| 野田 | リテンション | 非プロジェクト完了チャーンレート（月次） | 15%以下 | 四半期（月次モニタリング） |
| 川畑 | チーム全体 | 共通KPIのARR達成 | 共通KPIと同一 | 四半期 |
| 内田 | コンバージョン | トライアル→有償CVR | Q1：6% / Q2：9% / Q3：12% / Q4：15% | 四半期累計 |

### ARRとの構造的関係

```
共通KPI：来年度末スナップショットARR
  │
  ├─ 流入（ユーザーを増やす）
  │    ├ 新規有償化（トライアル→有償） ← 内田のCVRが効くレバー
  │    └ 既存顧客のシート拡大         ← チーム全体で対応
  │
  └─ 流出（ユーザーを減らさない）
       └ 解約（チャーン）              ← 野田のチャーン抑制が効くレバー

個別KPI：
  野田   → 非PJ完了チャーンレート ≤ 月次15%（流出を防ぐ）
  内田   → トライアル→有償CVR ≧ 四半期別目標（流入を増やす）
  川畑   → ARR達成（全体を統合する）
```

3人がそれぞれARRの異なるレバーを担当し、共通KPIのARRで「同じ船に乗っている」状態を作る構造です。

上記の定量KPIに加え、営業部の組織活性化に向けた定性目標（営業ノウハウのDB化、履歴登録の自動化、営業資料の改善、新機能の簡易要件定義など）にもチーム全体で取り組み、人事評価に反映します（詳細は「4. 定性目標」を参照）。

## 3. 各KPIの設計意図

### 3.1 共通KPI：来年度末スナップショットARR

#### 定義

```
ARR = 3月末時点の有償ユーザー数 × ユーザー単価 × 12
    = 3月末時点のMRR × 12

※ 途中の累計売上ではなく、末時点の残存ユーザーベースで算出する。
```

#### なぜARRを共通KPIにするのか

プロダクト立ち上げ期において、売上を最低限立てることは不可欠である。一方で、以下のシナリオは避けなければならない。

- (a) 売上は立っているが、チャーンが激しくユーザーが定着していない
- (b) チャーンが多いためにFSが新規獲得し続けなければならない状態が常態化する

ARRはストック指標であり、「新規獲得の勢い」と「既存ユーザーの定着」の両方が反映される。この1つの指標で全員が同じ方向を見ることで、3人チームの判断基準をシンプルに保てる。

#### 当初案（武井さんからの打診）との違い

当初、武井さんから打診されたKPIは「チャーンレート15%以下」と「ユーザー数460人」の2つであった。この2つを並列に置くことについて、以下の課題を感じている。

**ユーザー数460人について：** FS5名×92人の新規獲得目標の総和であり、本質的にはセールスの獲得目標である。CSのKPIとして持つと、未達時にFSとCSの責任が切り分けられなくなる。

**チャーンレートとユーザー数の並列について：** チャーンレートは「ユーザー数がどう増減するか」の内訳の1つ（減少要因）である。結果指標であるユーザー数と、その内訳であるチャーンレートを同じ階層に並べると、KPIの因果構造が見えにくくなる。

具体例で示す。来年度末にユーザー数が400人だった場合を考える。

- 「ユーザー数460人」のKPIは未達
- 「チャーンレート15%以下」のKPIは達成

この場合、「チャーンは抑えられているのにユーザーが足りない」のか「チャーンレートは15%以下だが、絶対数として多く離脱している」のかが即座に判断できない。ユーザー数という結果と、その構成要因であるチャーンを分けて管理する方が、原因分析と対策のスピードが上がる。

そこで、共通KPIはARRに一本化し、チャーンレートは野田の個別KPIとして管理する構造とした。

### 3.2 個人KPI：野田（リテンション）

#### KPI

```
非プロジェクト完了チャーンレート（月次） ≤ 15%

算出式：
（PJ完了以外の理由で解約したユーザー数 ÷ 月初時点の有償ユーザー数）× 100
```

#### 目標値

月次15%以下。収益シミュレーション上の設定値を採用。チャーンの過去実績がまだないため、まずこの数値をベースラインとして運用し、実績が蓄積された段階で見直す。

#### なぜ「率」で見るのか

絶対数（年間解約ユーザー数 ≤ N人）で管理する方法もあるが、率を採用した。

FSの新規獲得が想定通りに進まなかった場合、有償ユーザーの母数が小さいまま推移する。絶対数で管理すると、同じ「10人解約」でも母数200人のときと母数80人のときで意味がまったく異なる。率であれば、FSの獲得状況に左右されず、野田自身のリテンション活動の成果を正しく評価できる。

#### なぜ「非プロジェクト完了」に限定するのか

CiviLinkには、サービスに満足していてもプロジェクト終了に伴い契約を終了するケースがある。設計業務の完了、工事の完了など、顧客側の業務サイクルに起因する解約であり、CSの活動では構造的に防げない。

| 分類 | 内容 | CSで防げるか |
|---|---|---|
| 不満チャーン | プロダクトへの不満・期待値とのギャップで解約 | 防げる可能性がある |
| PJ完了チャーン | 業務終了に伴う計画的な契約終了 | 構造的に防げない |

この2つを混ぜてKPIにすると、野田がコントロールできない要因で数字が悪化する。KPIは「本人の活動で動かせる成果」で持つべきであり、PJ完了チャーンを除外することで、野田が注力すべきアクション（不満要因の解消、活用促進、早期アラート対応など）に集中できる環境を作る。

解約発生時には理由タグを記録し、PJ完了チャーンとそれ以外を切り分ける運用を導入する（詳細は「5. 運用ルール」を参照）。

#### なぜ行動量KPIを持たないのか

「有償ユーザーへの定期接点実施率」のような行動量指標をKPIにする案もあったが、採用しなかった。FSが「架電量」ではなく「売上」をKPIとし、架電量はそのための手段として自分で設計・振り返りするのと同じ考え方である。

チャーンレートという成果でKPIを持ち、そのために「何をするか（接点頻度、ヘルスチェック、活用促進の方法など）」は野田自身が設計する。KPIレベルで手段を固定すると、状況変化への対応が遅れ、運用が単調になるリスクがある。

### 3.3 個人KPI：川畑（チーム全体）

#### KPI

```
チーム共通KPIのARR達成
```

#### 考え方

FSマネージャーがチーム全体の売上目標を持つのと同じ構造で、共通KPIをそのまま個人KPIとして持つ。

野田がチャーンを抑え、内田がCVRを上げ、その全体をARR達成に繋げることが川畑の役割である。加えて、このフェーズでは自身もフィールドに出ることが必要であり、特定の変数に固定するよりも、状況に応じて柔軟に動ける状態を維持する。

#### 他の案を検討した経緯

当初は「既存顧客からのシート拡大数」「Net Expansion Rate」「NRR」を個別KPIとする案を検討した。しかし、有償顧客基盤が11社62ユーザーという規模では、1社の動きで指標が乱高下するため安定的に機能しない。前年度からの顧客基盤がほぼゼロに近い状況では、既存基盤の維持・拡大を測る指標は時期尚早と判断した。

### 3.4 個人KPI：内田（コンバージョン）

#### KPI

```
トライアル→有償CVR（四半期別目標）

算出式：
（トライアルから有償に転換したユーザー数 ÷ トライアル終了ユーザー数）× 100

四半期別目標：
  Q1（4-6月）：6%
  Q2（7-9月）：9%
  Q3（10-12月）：12%
  Q4（1-3月）：15%
```

#### 目標値

FS1人あたり年間売上目標600万円を達成するための収益シミュレーションから逆算された数値である。四半期ごとに段階的に上昇する設計は、プロダクトの機能拡充やCSチームの習熟度向上を織り込んでいる。FS側のアクション量設計と連動しており、内田のCVR達成がFS側の売上達成の前提条件となっている。

#### なぜCVR一本なのか

内田の業務は商談支援、営業資料整備、オンボーディング、VOC回収→要件定義→開発トスアップと幅広い。一見すると1つのKPIでは捉えきれないように見えるが、これらはすべて「トライアルユーザーが有償化するかどうか」に影響を与える活動である。

- 商談支援で顧客の期待値を正しく設定すれば、トライアル後のギャップが減りCVRが上がる
- 営業資料の質が高ければ、トライアル開始時の理解度が上がりCVRが上がる
- オンボーディングが良ければ、プロダクトの価値実感が早まりCVRが上がる
- VOCを活かしてプロダクト改善を推進すれば、中長期的にCVRが上がる

FSが「売上」という1つの成果KPIを持ち、そのために商談・提案・クロージング・資料作成など多様な手段を使うのと同じ構造である。CVR一本にすることで、「何に時間を使うか」の優先順位を内田自身が判断できる自由度を確保する。

#### 他の案を検討した経緯

- **VOC回収率：** 行動量KPIに近く、「回収したが、そこから何が改善されたのか」が問われない。運用が単調になるリスクがある。
- **VOC起票→開発トスアップのリードタイム：** 現状、回収からトスアップまではそれほど時間がかからず、トスアップ後の判断は開発部の裁量のため、CS側の責任範囲が曖昧。
- **商談→トライアル開始CVR（2段階CVR）：** 内田の商談支援の貢献を測れるが、FS側の要因も大きく、内田単独のコントロール範囲を超える。

#### なぜ四半期累計で評価するのか

トライアル開始数はFSの獲得力に依存するため、月単位では分母が小さく、CVRが大きく振れる可能性がある。月次はモニタリングに留め、評価は四半期累計とすることで安定した指標運用ができる。

## 4. 定性目標：営業部の組織活性化施策

### 位置づけ

前述の定量KPI（ARR・チャーンレート・CVR）に加え、営業部を組織として活性化させるための施策にもCSチーム全体で取り組む。これらは直接的な数値KPIには反映されにくいが、FSを含む営業部全体の生産性向上・基盤強化に不可欠な取り組みであり、人事評価における定性目標として各メンバーのKPIに組み込む。

### 想定施策

| # | 施策 | 内容 |
|---|---|---|
| 1 | 営業ノウハウの蓄積・データベース化 | 商談で得た知見や成功パターンをチームで再利用可能な形で蓄積する |
| 2 | FS向けリードリスト消化履歴の自動化 | 架電結果やヒアリング内容を確実に記録できる仕組みを構築し、FSの業務負荷を軽減する |
| 3 | 営業資料のブラッシュアップ | CiviLinkの提案資料・トライアル説明資料などを継続的に改善する |
| 4 | 新機能の簡易的な要件定義 | VOCや市場動向をもとに、開発部へのインプットとなる簡易要件を整理する |

### 担当の考え方

現時点では、誰がどの施策を主担当するかは厳密に整理できていない。来年度の活動を通じて、各メンバーの得意領域や業務負荷を見ながら分担を決めていく。定量KPIと同様に、四半期ごとの振り返りの中で進捗と貢献度を確認し、人事評価に反映する。

## 5. 運用ルール

### 5.1 解約理由タグの記録

解約が発生した際は、以下の理由タグを記録する。タグの判断は野田が行う。

| タグ | 定義 |
|---|---|
| プロダクト不満 | 機能不足・バグ・UIなど、プロダクト起因の解約 |
| 費用対効果 | 価格に見合わない、ROIが合わないと判断された解約 |
| プロジェクト完了 | 業務終了に伴う計画的な契約終了。サービス自体への不満はない |
| 社内事情 | 予算凍結・担当者異動・組織変更など、顧客側の内部要因 |
| その他 | 上記に該当しない解約 |

PJ完了と不満が重なる複合的なケース（プロジェクトは完了したが、継続案件があるにもかかわらずプロダクトへの不満で解約するなど）の判断は野田に一任する。

### 5.2 モニタリングと評価のサイクル

| サイクル | 内容 |
|---|---|
| 月次 | 全指標（チャーンレート・CVR・ARR進捗）をモニタリング |
| 四半期 | 個別KPIの正式評価 |
| 日次・週次 | 各メンバーが自身のアクション計画を振り返り |

### 5.3 KPIと手段（How）の分離原則

本KPI設計の基本思想として、KPIは成果指標（What：何を達成するか）で設定し、手段（How：どうやって達成するか）は各メンバーが自ら設計する。

FSが「売上8,500万円」という成果目標を持ちながら、そのための架電量や訪問計画は自分で組み立てるのと同じ考え方である。マネージャーは手段の設計を支援するが、KPIレベルで手段を固定しない。
