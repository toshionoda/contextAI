**製品方針（重要）**: **自治体ごとのスキーマ・テンプレを持たない**。「図面PDFを投入すると数量が抽出される」汎用ロジックを中核に置き、自治体差は出力フォーマッタ（数量計算書PDF / Excel総括表 / 汎用JSON）で吸収する。中間表現はJS（日本下水道事業団）共通工種体系を基準にする。

**書式パターン参照**: JSの「下水道施設CAD製図基準」が全国97pの標準仕様を定めており、レイヤ命名規則4階層・ファイル命名規則・図面種別CODE BOOKまで統一。政令市（名古屋市等）の完成図作成基準も福岡市と記載項目レベルで一致。**この共通性は「テンプレを書けば吸収できる」根拠ではなく、「中間表現をJS体系に正規化すれば全国で再利用可能」根拠として活用**。詳細は `customers/municipal/comparison/_pattern_analysis.md`

**推奨アクション**: **2-3週間の「事業性追加調査フェーズ＋社内アセット連携設計」を経てから本格Goを判断**。
- 調査ピン: ①競合の下水道参入動向 ②政令市5-7社へのニーズ深掘り ③社内アセット所有者との連携設計（リソース・優先度・統合方式）
- これら3点で事業仮説が立証されればGo（社内アセット活用で6/9週からPhase 1着手可能）、否定されればパートナー連携 or 紹介に切り替え。

## 1. 問題定義（相談背景の整理）

### 1.1 相談の性質（重要・前提）
- **発信元**: mirai@（未来アット）でDXを担当している方。**元下水道部署所属**で、古巣の業務ニーズを共有してくれた
- **段階**: 「実証PoCをやってください」ではなく **「製品があったら使いたい」**
- **意味**: マルメ側の主導判断（自社プロダクトとして開発するかどうか）が問われている
  - 福岡市は **製品ができれば最初の顧客になる見込みあり**（mirai@経由の実証フィールド付き）
  - ただしマルメが作らなければ、別ベンダー（東芝等）の製品を採用する可能性も高い

### 1.2 想定される業務フロー（顧客側）
```
[CAD図面（平面・縦断・横断）]
    ↓ 数量拾い（人手・図面読み取り）          ← 最も人手依存
[数量計算書（路線ごとの計算過程）]
    ↓ 集計
[数量総括表（工種L2-L3-L4階層）]
    ↓ 単価適用
[金抜き設計書 → 金入り設計書]
    ↓ 発注（電子調達）
```

### 1.3 製品が解決すること
- 図面PDF → 数量計算書ドラフトの自動生成（精度70-85%、**書式に依存せず汎用抽出**）
- 数量総括表への自動集計（JS共通工種体系で正規化、L2-L3-L4階層）
- 整合性チェック（図面と数量の不一致検出）
- 自治体差は **出力フォーマッタ** でのみ吸収（書式・単価コード体系・列順序）
- **PoCでは DWG 入力は対象外**（PDF限定）

## 2. サンプル資料分析（5本の関係構造）

### 2.1 文書間の依存関係
```
[図面一式（DWG → PDF）] ──┐
                          ├─→ [数量計算書①②] ──→ [数量総括表] ──→ [金抜き設計書]
[管路台帳・歩掛・標準仕様] ──┘    （路線ごと計算）   （工種階層集計）   （単価適用）
```

### 2.2 重要観察
1. **本案件は管更生工が主体** = 既設下水道改築。新設より自動化親和性高い（管路情報が下水道台帳DB化済み）
2. **書式は全国で類似だが、自治体ごとに単価コード体系・条件記法が独自**（福岡市はB07GH/PG/CB/ZD/PD体系）
3. **福岡市は下水道台帳デジタル化済み**（2026年4月 立会受付Webシステム運用開始）→ 図面ベースに加えて **「台帳→計画路線→自動計算」の別アプローチも成立**
4. **数量計算書のフォーマットは極めて定型化**（路線×工種×規格のクロステーブル）= 自動化親和性高い

## 3. 技術実現性の評価

### 3.1 自動化対象の3階層

| 段階 | 内容 | 自動化難易度 | 既存ソリューション |
|---|---|---|---|
| **A) 図面 → 数量計算書** | 数値・規格認識 | **高**（コア技術課題） | AI積算、東芝CISSART |
| B) 数量計算書 → 数量総括表 | 工種階層集計 | 低 | 多数の積算ソフトで対応 |
| C) 数量総括表 → 金抜き設計書 | 単価マッピング | 中 | 積算王、APRO、ARES、頂等 |

### 3.2 入力形式別の方針（PoC）

| 入力 | PoC対応 | 備考 |
|---|---|---|
| **PDF（ベクター）** | **★ 主対象** | LLM Vision + ベクター構造化で汎用抽出 |
| **PDF（ラスタ／スキャン）** | △ 二次対象 | OCR前処理を挟む。精度は劣る前提 |
| **DWG** | **× PoC対象外** | DWGパーサーTrack A完成後に発展課題として再検討 |
| **下水道台帳DB** | × | 別アプローチ。今回スコープ外 |
| 紙図面 | × | 不要 |

### 3.3 推奨技術アプローチ（汎用抽出方針）

**コア原則**:
- **入力側にスキーマを持たない** — 自治体ごとのレイヤ命名規則・タイトル枠フォーマット差を吸収するルールを書かず、Vision LLM で読む
- **中間表現は1つ** — JS共通工種体系（足りないフィールドは独自拡張）に正規化
- **書式差は出力レンダラーでだけ吸収** — 単価コード・列順序の違いはテンプレでなくフォーマッタで処理

**処理フロー（PoC）**:
```
[PDF（図面）]
  → ベクター構造化（社内アセット①②）
  → Vision LLM で工種・規格・数量を直接読取
  → JS共通工種体系で正規化（中間JSON）
  → 出力レンダラー（自治体PDF / Excel総括表 / 汎用JSON）
```

**段階的実装**:
1. **PoC（〜3-4ヶ月）**: PDF限定 / 福岡市1件で抽出ロジックを検証 → 精度70-85%
2. **Phase 2（〜9-13ヶ月）**: 出力フォーマッタ汎用化 + 他工種拡張 + 精度改善
3. **発展課題（PoC対象外）**: DWG対応（Track A完成タイミング）／自治体台帳API連携ハイブリッド

**注意点**: 完全自動化は不可。AI出力をドラフト、人手で根拠書類化が業界共通認識。

## 4. 市場・競合分析

### 4.1 既存競合（参入済み）

| サービス | 対応領域 | 強み | 下水道対応 | 評価 |
|---|---|---|---|---|
| **東芝CISSART** | 道路・河川中心 | 数量計算書・総括表・整合性チェック・正誤表まで自動。i-Construction北陸地整採択 | **未明示**（拡張可能性高） | **最強の潜在競合** |
| AI積算（CADジャパン） | 建築設備（空調・衛生・消火） | PDF対応、個数物99%・ルート系95% | 主領域外 | 直接競合ではない |
| 積算AI（KK Generation） | 国内特化 | 詳細未確認 | 不明 | 要追加調査 |
| 積算王／APRO／ARES／頂／明積／ランド2000 | B→C段階（数量入力→設計書） | 自治体実績豊富 | あり | A段階を持たない=直接競合外 |

### 4.2 競合優位の窓（2-3年と推定）

- **東芝CISSARTが下水道領域に本格参入する確率: 中〜高**
  - 道路河川で実績あり、自治体顧客基盤を持つ
  - 横展開の自然な次フィールド
- **CiviLinkの先行優位窓: 2-3年**
  - その間に下水道特化プロダクトとして実績を作れるかが勝敗を分ける
  - 競合追従後は「価格 + プラットフォーム連携」で戦う必要

### 4.3 政策的追い風

- 国交省 i-Construction 2.0: 2026年マイルストーンで数量算出自動化を推進
- 公的調達・実証予算の窓が開いている（mirai@、スタートアップ公共調達サポート事業）
- 管路更新需要：全管路更新に150年要する低更新率 → 改築需要は今後30年爆発（継続的発注）

## 5. 事業性評価（トップダウンショート）

### 5.1 市場規模試算

#### TAM（全国上下水道事業者）
- 全国の下水道事業者：1,500市町村 + 47都道府県 ≒ **1,550事業体**
- 1事業体あたり年間下水道工事発注：平均10件 × 平均積算工数20時間 = 200時間/年
- 全国合計：約 **31万時間/年**
- 単価¥1万/時間相当 → **TAM約31億円/年**

#### SAM（業務量が大きく予算もつく層）
- 政令市20 + 中核市62 + 特例市40 ＝ **約120事業体**
- 1事業体年間50件 × 30時間 = 1,500時間/年
- 120事業体 × 1,500時間 = **18万時間/年**
- 業務時間ベース：**約18億円/年**
- SaaS料金ベース：年¥3-5M × 120事業体 = **¥3.6-6億円/年（ARR天井）**

#### SOM（5年で取れる現実的な規模）
- 5年で**30事業体**獲得（先進政令市中心、福岡発全国波及）
- 年¥3M × 30事業体 = **ARR約¥9,000万円/年**
- 5年ARRピーク：**¥6,000万-1.5億円**（シナリオ次第）

- 社内アセット流用により、Worst Caseでも4年目で投資回収可能（ゼロからだと達成不可）

### 5.4 競合優位性（差別化軸の整理）

| 軸 | 内容 | 競合優位の持続性 |
|---|---|---|
| **下水道工事への特化** | 工種・歩掛・更生工法の深い対応 | 中（東芝が下水道に来たら侵食される） |
| **CiviLinkプラットフォーム連携** | 過去案件参照・台帳統合・計画→施工→台帳更新の循環 | **高**（CiviLink自体の堀次第） |
| **社内アセット蓄積（要素技術6本）** | 図面PDF構造化・レイアウト検出・章立て解析・表抽出・整合性確認・エージェント設計 | **高**（数十人月分の蓄積を競合は持っていない） |
| **書式に依存しない汎用抽出精度** | スキーマ／テンプレを持たず、Vision LLM＋JS共通工種体系で読む | **中-高**（データ蓄積で精度差が広がる構造） |
| **mirai@発の自治体DXストーリー** | 福岡市実証→全国広報の経路 | 中（ローンチ初期のブースト） |
| **3者連携モデルの発注DX** | 自治体×補修設計×地場建設 のレイヤー位置付け | 高（CiviLinkの戦略文脈と整合） |

**結論**: 単独機能としての差別化は中程度。**CiviLink本体（プラットフォーム連携）＋社内アセット蓄積＋汎用抽出のデータフライホイール の3点で売る前提なら高い差別化が成立**。

### 5.5 リスクシナリオ

| リスク | 影響 | 対策 |
|---|---|---|
| **東芝CISSARTが下水道に参入** | Worstケース化 | 2年で実績10社を作り先行優位確立 |
| **i-Construction 2.0で標準化が一気に進む** | 各社のアプローチが収斂し差別化困難 | 自治体パッケージ全体（3者連携）で売る |
| **汎用抽出の精度が想定（70-85%）に届かない** | ドラフト品質が低く人手修正コストが残る | Phase 1で福岡市1件＋もう1自治体のPDFで精度実測。Vision LLM＋社内アセット組合せで届かなければアセット5（drawing-calc-matcher、F1=74.9%実績）の手法を本線に切替 |
| **PDF（ベクター）以外の入力比率が想定より高い** | スキャンPDF・DWGネイティブ要件が多発 | PoCで入力形式の比率を実測。スキャン比率高ければOCR前処理を厚く、DWGネイティブ要件多ければTrack A完成タイミングを早める検討 |
| **福山ピン1への工数侵食** | CiviLink本体PMF達成遅延 | 開発工数は完全別チーム化、野田さん工数は事業企画のみ |

### 5.6 「事業性が見込めるか」の判断軸（5点）

| # | 判断軸 | Base想定での見立て | Goの閾値 |
|---|---|---|---|
| 1 | 5年累計売上 | ¥1.9億 | **¥1.5億以上** |
| 2 | 投資回収期間 | 3年 | **3年以内** |
| 3 | LTV/CAC | 6倍 | **3倍以上** |
| 4 | 競合優位の窓 | 2-3年 | **2年以上** |
| 5 | CiviLink本体への影響 | 工数別管理で限定的 | **福山ピン1進捗に影響しない** |

**Base Caseで5項目すべてクリア → 条件付きGo**

## 6. 戦略整合（CiviLink拡張モジュール前提）

### 6.1 既存戦略との整合性

| 戦略 | 関係 | 整合性（拡張モジュール前提で再評価） |
|---|---|---|
| **CiviLink本体（福山ピン1集中）** | 顧客セグメント別／業務領域別 | **△ 工数別管理が必須**（野田さんは事業企画のみ、開発は別チーム） |
| **DWGパーサーPJ（Track A）** | 図面→数量の中核技術 | **○ シナジー（PoC後）**（PoCはPDF限定。Track A完成後にDWG入力対応で精度・適用範囲拡大） |
| **自治体3者連携モデル** | 発注DX・補修設計効率化 | **★ 整合**（モデルの「発注DX」レイヤーに位置付く） |
| **戦略書「やめる3つ」** | コンセプト共感案件追跡をやめる | **〇** 製品開発の意思決定なので「コンセプト共感」とは異なるカテゴリ |
| **戦略書「業務診断セッション」** | 売り手モード脱却 | 〇 自治体との対話は「業務診断」スタイルで継続 |
| **2026羅針盤（5セグメント・ACV4700万）** | 自治体セグメントの収益貢献 | **★ 整合**（自治体パッケージの収益化エンジンになり得る） |

### 6.2 「拡張モジュール扱い」が解く問題

- **野田さんの工数分散リスク**: 開発を別チーム化すれば事業企画のみで済む（週1-2日）
- **福山ピン1への影響**: モジュール開発はCiviLink本体PMFと並行・独立進行可能
- **戦略書「やめる3つ」との衝突回避**: 製品開発は「全国デモ行脚」「コンセプト共感案件追跡」と異なるカテゴリ
- **DWG Track Aとのシナジー創出**: モジュールはTrack Aの最初の商用応用先になる

## 7. Go/No-Go判断と次アクション

### 7.1 推奨アクション: **2-3週間の事業性追加調査＋社内アセット連携設計フェーズを挟む**

トップダウン試算は仮説の塊なので、本格Go判断の前に **3つの調査ピン** を潰す:

#### ピン①: 競合の下水道領域参入動向（先行優位の窓を確定）
- 東芝CISSART営業に直接ヒアリング（下水道対応有無・ロードマップ）
- 北陸地整以外の自治体実証案件を追跡
- AI積算・KK Generation積算AIの下水道対応状況確認
- **判断軸**: 2-3年の窓が確認できれば Go方向

#### ピン②: 政令市5-7社へのニーズ深掘り（市場性検証）
- 福岡市以外の政令市（北九州、神戸、その他）にニーズヒアリング
- 想定単価¥3M/年 が現実的か検証
- 3者連携モデル議論中の佐賀・北九州・福岡で同時にニーズ存在するか確認
- **判断軸**: 5社中3社以上で「製品があれば使いたい」+ ¥3M/年が許容範囲なら Go方向

#### ピン③: 社内アセット連携設計（実装現実性）
- 6本のアセット所有者・主担当者を特定し、福岡市案件への流用可否を協議
  - rd-pdf-component-extractor / rd-blueprint-layout-detection-engine / rd-rebar-drawing-hierarchy / rd-tekkin_hyo_chushutsu / rd-poc-drawing-calc-matcher / drawing_LLM_lab
- 並列案件（**152_kengi-kotei-gaisan**＝建技 大西さん／**116 オリエンタルコンサルタンツ**地下構造物図面）との統合・分業設計
- アセット商用化の権利関係（顧客先案件由来のIP・ライセンス）を確認
- Phase 1 PoCの具体スコープ・PMアサイン・工数見積（社内アセット流用で2-3人月の妥当性、**入力PDFのみ**前提）
- 中間表現（JS共通工種体系＋拡張フィールド）の設計合意
- **判断軸**: 6本のアセットのうち4本以上が利用可能、累計¥900-1,650万の予算化が現実的、なら Go方向
