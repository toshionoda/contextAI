# 佐賀市 インフラ長寿命化DX提案：橋梁から全インフラへ

**作成日:** 2026-04-14
**対象:** 佐賀市デジタル推進課
**提案者:** 株式会社マルメ（伊藤忠商事DXパートナー）
**キーメッセージ:** 橋梁は入り口にすぎない。佐賀市が管理する全てのインフラに、同じDX基盤を横展開する。

---

## 1. デジタル推進課への問いかけ

> 佐賀市のインフラ維持管理において、「デジタルの力で本当に変えられること」は何か？

佐賀市DX推進方針（令和4年策定）は、行政サービスのデジタル化を掲げています。しかし、**インフラ維持管理のDX**は、まだ手つかずの巨大な領域です。

本提案は、橋梁のPOCで実証した仕組みを、**道路・下水道・排水施設・公園・トンネル**全てに展開し、佐賀市のインフラ管理を根本から変えるものです。

### 共通する3つの構造的課題

```
課題① 老朽化が加速している
  橋梁: 35%が築50年超 → 30年後には96%が築50年超
  下水道: 昭和53年(1978年)供用開始 → 築48年、管渠老朽化が進行
  排水施設: 高度成長期に整備された排水機場の更新時期が到来

課題② 人が足りない
  全国の市区町村の約4割が技術系職員5名以下
  担当者が異動すると「あの施設の履歴は○○さんしか知らない」
  予防保全に必要な高度な判断ができる人材が不足

課題③ データがバラバラ
  橋梁は橋梁台帳、道路は道路台帳、下水は管渠台帳、公園は公園台帳…
  部署ごと・施設種別ごとにExcel・紙・独自システムで分断
  横断的な優先順位判断ができない
```

## 3. デジタル推進課に刺さるストーリー：「データが繋がると、判断が変わる」

### Before: 部署ごとの縦割りインフラ管理

```
道路整備課    → 橋梁台帳（Excel/紙）      → 橋の補修を個別判断
道路管理課    → 道路台帳（紙/GIS）        → 舗装修繕を個別判断
上下水道局    → 管渠台帳（独自DB）        → 管更新を個別判断
河川砂防課    → 排水機場台帳（紙）        → 排水機場更新を個別判断
緑化推進課    → 公園台帳（Excel）         → 遊具更新を個別判断
     ↓
【結果】同じ区間で橋の補修と舗装修繕が別年度に発注される
       → 2回通行止め、2回の設計費、2回の交通規制費
```

### After: インフラデータ基盤で横断管理

```
                    ┌─────────────────────────┐
                    │   CiviLink（インフラデータ基盤）  │
                    │                           │
  道路整備課 ─────┤  橋梁 ← 同一区間 → 舗装     │
  道路管理課 ─────┤  トンネル ← 関連 → 道路     │
  上下水道局 ─────┤  管渠 ← 埋設位置 → 道路     │
  河川砂防課 ─────┤  排水機場 ← 流域 → 河川     │
  緑化推進課 ─────┤  公園遊具 ← 位置 → 地図     │
                    │                           │
                    │  → 横断的な優先順位判断      │
                    │  → 同一区間の工事集約        │
                    │  → 部署を超えた技術継承      │
                    └─────────────────────────┘
```

**デジタル推進課にとっての価値:**
- 「DXを各部署に個別導入」ではなく、**「1つの基盤を全庁横断で活用」**
- 行政マネジメント課の公共施設等総合管理計画の**実行ツール**になる
- 佐賀市DX推進方針の**具体的な成功事例**になる

### 4-2. 道路（舗装）

**現状の課題:**
- 都市計画道路90路線145.5km + 市道（総延長は要確認）の舗装管理
- 損傷の発見はパトロール・住民通報に依存。**「走るだけで分かる」技術が存在**
- 舗装長寿命化修繕計画は総合管理計画に包含されているが、**個別計画は未策定の可能性**

**DX提案:**

| メニュー | ツール | 効果 |
|---------|-------|------|
| **AI路面診断** | みちログ（アイシン）or マルチファインアイ（福田道路） | 車両走行だけで路面異常を自動検出。専用機材不要 |
| **路面データのCiviLink集約** | CiviLink | AI検出結果を道路図面上にマッピング。経年変化を可視化 |
| **修繕優先順位のデータ化** | CiviLink + Excel連携 | 損傷度×交通量×予算で優先順位を自動算出 |
| **舗装長寿命化修繕計画の策定支援** | マルメFDE | 計画未策定であれば、データ収集→計画策定を一貫支援 |

**デジタル推進課への訴求:**
> 「道路パトロール車にスマホを載せるだけで、市道全線の路面状態が自動記録される。
> これを橋梁の点検データと同じCiviLink上で見れば、
> 『この区間は橋も道路も傷んでいる。まとめて補修しよう』という判断ができる。」

---

### 4-3. 下水道

**現状の課題:**
- 昭和53年（1978年）供用開始 → **築48年**。全国的に管渠の老朽化が深刻
- 汚水ポンプ場7箇所の機械設備も更新時期
- 資本的収支は**年間▲15.7億円**（内部留保で補填中）
- ストックマネジメント計画（R2策定）で予防保全への転換を志向

**DX提案:**

| メニュー | ツール/手法 | 効果 |
|---------|-----------|------|
| **管渠CCTV調査データの一元管理** | CiviLink | 調査映像・損傷写真を管渠図面に紐付け。過去調査との比較が容易に |
| **マンホール蓋の劣化診断** | スマホ撮影 + AI画像解析 | 巡回時にマンホール蓋を撮影→劣化度を自動判定 |
| **ポンプ場の点検記録デジタル化** | CiviLink + チェックリストTP | 7箇所のポンプ場点検を紙→デジタル。異常値を自動アラート |
| **管渠更新優先順位の可視化** | CiviLink + GIS連携 | 管齢×損傷度×重要度でカラーマップ表示 |

**デジタル推進課への訴求:**
> 「下水道管は地下に埋まっていて『見えない』インフラ。だからこそデータ管理が命。
> 築48年の管渠がどこから壊れるか、CCTV調査のデータを蓄積すれば予測できるようになる。
> 管渠図面の上に調査結果を重ねて見る — これは橋梁で図面に点検写真を紐付けるのと全く同じ仕組み。」

---

### 4-4. 排水施設（佐賀市特有の最重要課題）

**現状の課題:**
- 佐賀平野の**低平地特有の地形**。自然排水が困難で排水機場・水門に依存
- 排水対策基本計画：64mm/hr降雨に対し浸水家屋面積**50%削減**が目標
- 第2回改訂（R7）で佐賀駅周辺400haに**14,000トンの追加貯留**目標を追加
- 排水機場の運転は**出水期の手動操作**が基本。操作員の高齢化も課題

**DX提案:**

| メニュー | ツール/手法 | 効果 |
|---------|-----------|------|
| **排水機場の運転記録デジタル化** | CiviLink + IoTセンサー | 水位・運転時間・燃料消費を自動記録。紙日報を廃止 |
| **排水機場設備の点検記録管理** | CiviLink + チェックリストTP | 機械設備の劣化状況を図面紐付きで蓄積 |
| **水門・樋門の遠隔監視** | IoTセンサー + クラウド | 出水期の現場急行を減らし、遠隔で状況把握 |
| **浸水シミュレーションデータの統合** | CiviLink + GIS | 排水能力×降雨予測×地形で浸水リスクを可視化 |

**デジタル推進課への訴求:**
> 「佐賀市にとって排水は命綱。毎年の出水期に排水機場を動かす判断は、ベテラン操作員の経験に依存している。
> その判断パターンをデータとして残す仕組みが、CiviLinkにはある。
> 操作員が引退しても、過去の水位×降雨量×ポンプ運転の履歴があれば、次の世代が判断できる。
> これは橋梁のベテラン技術者の判断ログを残すのと、全く同じコンセプト。」

### 4-5. 公園施設

**現状の課題:**
- 都市公園71 + 児童遊園19 + 小公園460+ = **550箇所以上**の公園を管理
- 遊具の老朽化 → 子どもの安全に直結。社会資本整備計画Phase 3で計画的更新を推進
- 巡回点検は目視が基本。点検結果は紙ベースで記録

**DX提案:**

| メニュー | ツール/手法 | 効果 |
|---------|-----------|------|
| **遊具点検のデジタル化** | スマホ + CiviLink | 遊具をスマホ撮影→点検チェックリストに図面紐付きで記録 |
| **公園施設台帳のクラウド化** | CiviLink | 550+箇所の公園施設情報を一元管理。検索・更新が即時 |
| **住民通報の図面紐付け** | LINE WORKS Bot + CiviLink | 住民からの「遊具が壊れている」通報を公園図面上にプロット |
| **修繕優先順位の可視化** | CiviLink | 遊具の種類×設置年数×利用頻度で更新優先度をスコアリング |

**デジタル推進課への訴求:**
> 「公園は市民との接点が最も近いインフラ。遊具の故障を住民がLINE WORKSで通報すると、
> CiviLink上の公園図面に自動でピンが立つ。緑化推進課の担当者はスマホで確認して、
> その場で対応状況を更新できる。市民にとっても『通報したらちゃんと直った』が見える。
> 市民体験の向上とインフラ管理の効率化が同時に実現する。」

## 5. 「1つの基盤で全インフラ」が実現する理由

### なぜCiviLinkが横展開できるのか

| インフラ種別 | 管理の本質 | CiviLinkでの対応 |
|------------|----------|----------------|
| 橋梁 | **図面**に対して点検結果を紐付ける | 図面ビューア + コメント + 承認WF |
| 道路 | **路線図**に対して損傷位置を紐付ける | 同上（図面 = 路線図） |
| 下水道 | **管渠図面**に対して調査結果を紐付ける | 同上（図面 = 管渠図面） |
| 排水施設 | **設備図**に対して点検記録を紐付ける | 同上（図面 = 設備配置図） |
| 公園 | **公園図面**に対して遊具状態を紐付ける | 同上（図面 = 公園配置図） |
| トンネル | **展開図**に対して損傷を紐付ける | 同上（図面 = トンネル展開図） |

**全てのインフラ管理の本質は同じ:**
> **「図面の上に、いつ・誰が・何を確認し、どう判断したかを記録する」**

CiviLinkはこの本質をSaaS化したもの。インフラの種類が変わっても、**図面がベースである限り同じ基盤で管理できる。**

## 8. なぜ今、デジタル推進課が動くべきか

### ①公共施設等総合管理計画の「実行フェーズ」に入った

佐賀市公共施設等総合管理計画（第2版）は計画を策定済み。しかし**実行のためのツールが未整備**。各部署がExcel・紙で個別管理している限り、計画は「絵に描いた餅」になる。

### ②国の制度が追い風

| 制度 | 内容 |
|------|------|
| i-Construction 2.0 | 2026年度からBIM/CIM標準適用。3Dモデル数量の直接積算利用を整備 |
| NETIS / 点検支援技術カタログ | AI点検技術を採用すれば工事成績評定で加点 |
| デジタル化・AI導入補助金 | 2026/3/30受付開始。クラウド利用料も最大2年分対象 |
| 佐賀県建設DX加速化補助金 | 2/3補助。ICT機器・ソフトウェアが対象 |

### ③佐賀市の「DX先進自治体」としてのブランディング

- **包括委託（2027年4月開始予定）**に向けて、DX基盤を先行整備することで、全国の自治体モデルになりうる
- 伊藤忠商事との連携で、佐賀市の事例を全国に横展開する体制がある
- デジタル推進課が旗振り役になれば、**「佐賀市型インフラDX」**として全国に発信できる
