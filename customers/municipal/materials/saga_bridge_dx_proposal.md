# 橋梁の調査・設計・施工 DX導入提案

**作成日:** 2026-04-13
**対象:** 佐賀市デジタル推進課 / 佐賀市契約のCランク以下地場建設会社
**提案者:** 株式会社マルメ（伊藤忠商事DXパートナー）
**ステータス:** ドラフト（4/20佐賀市訪問用）

---

## エグゼクティブサマリ

橋梁の維持管理は「**調査（点検）→ 設計（補修設計）→ 施工（補修工事）**」の3フェーズで進む業務です。各フェーズで大量のドキュメントと、自治体⇔施工会社⇔設計者の間でのやり取りが発生し、これが地場建設会社・自治体監督員の双方にとって最大の負担になっています。

本提案では：

1. **既存ICTツール（写真アプリ・チャット・クラウド等）のパッケージ導入**で、各フェーズの「点」の効率化を実現
2. **CiviLinkがそれらを横串で束ねる「ドキュメント管理ややり取りの基盤」**として、フェーズ間のデータ受け渡しを一気通貫化
3. 段階的導入と補助金活用で**地場建設会社の実質負担を最小化**

これにより、佐賀市の包括委託（2027年4月開始予定）に向けた1年間のPOC期間を、地場建設業界のDX定着の足がかりとします。

## 1. 佐賀市の橋梁維持管理が直面する3つの壁

### 壁①: 2024年問題（残業できない時代）
- 2024年4月から建設業にも時間外労働の上限規制（月45時間・年360時間）が適用
- 現場後の点検調書作成・補修報告書作成・写真整理が残業の温床
- **書類作業の効率化が経営課題に直結**

### 壁②: 技術者の高齢化と人材不足
- 建設業就業者の37%が55歳以上、29歳以下はわずか12%
- 2035年には技術者約3.2万人が不足見込み
- 「あの図面どこ？」「あの判断は○○さんしか知らない」が日常
- **属人化したノウハウのデジタル化**が急務

### 壁③: i-Construction 2.0への対応
- 2026年度から舗装工・地盤改良工でICT施工が原則化（佐賀市の補修工事に直撃）
- 電子納品の対応範囲も拡大中
- 自治体公共工事でのデジタル対応力が**受注競争力に直結**

### 特にCランク以下企業の事情
- IT/DX担当者が不在、Excel + Word + ローカルフォルダ管理が中心
- クラウド利用率は低く、「何から始めればいいか分からない」
- 「IT予算500万円未満」が約4割
- アンドパッド・フォトラクション等の有名ツールは**建築向け中心**で、土木・橋梁維持管理向けが手薄

## 2. 「調査・設計・施工」で発生するドキュメントと課題

橋梁1橋の維持管理サイクルで発生する主なドキュメントとやり取り：

### フェーズ① 調査（点検）
| ドキュメント | 現状の課題 |
|------------|-----------|
| 点検計画書、巡回ルート図 | 紙ベース、現場で持ち運び困難 |
| 損傷写真（数百〜数千枚/橋） | スマホ・デジカメで撮影後、PCに転送→手動仕分け |
| 損傷展開図、点検調書 | 紙の点検野帳→事務所で清書→Excel入力 |
| 自治体への点検報告書 | メールで提出、紙でも提出 |

### フェーズ② 設計（補修設計）
| ドキュメント | 現状の課題 |
|------------|-----------|
| 補修設計図面（CAD） | 個人PCにバラバラ保管、最新版が分からない |
| 数量計算書、設計計算書 | 図面との整合確認が手作業 |
| 自治体への照査依頼・回答 | メール添付＋紙ファイル、履歴が追えない |
| 設計変更指示・回答 | FAX・メール・電話で錯綜 |

### フェーズ③ 施工（補修工事）
| ドキュメント | 現状の課題 |
|------------|-----------|
| 施工計画書、特記仕様書 | 紙ベース、変更管理が難しい |
| 工事打合簿（数十枚/工事） | 紙→FAX→押印→PDF化→メール |
| 段階確認書、材料承認願 | 自治体への持参・FAX、確認待ちで工程遅延 |
| 工事写真（数千枚/工事） | 蔵衛門等で管理しても他資料と紐付かない |
| 出来形管理図表、品質管理書類 | Excel手入力、規格値チェックは目視 |
| 電子納品成果物 | 工事完了時にまとめて作成、形式エラー多発 |

### 共通する3つの構造的問題
1. **書類が散在**: フォルダ・USB・紙キャビネット・メールに散らばる
2. **やり取りが断絶**: 自治体⇔施工会社⇔設計者の連絡がメール・FAX・電話・対面に分散
3. **データが流れない**: 調査データが設計に活きず、設計データが施工に活きない（毎フェーズで再入力）

## 3. DXパッケージ導入：既存ツールですぐ始められること

各フェーズで、地場建設会社がすぐ使える既存ICTツールをパッケージで導入します。**新しいものを開発するのではなく、既にある良いツールを組み合わせる**のが基本方針です。

### 推奨ツールパッケージ

| カテゴリ | 推奨ツール候補 | 月額目安 | 用途 |
|---------|-------------|--------|------|
| **チャット** | LINE WORKS / Microsoft Teams | 1人500〜800円 | 現場⇔事務所⇔自治体の日常連絡 |
| **クラウドストレージ** | Google Workspace / Microsoft 365 | 1人800〜1,500円 | 一般文書・写真の共有保管 |
| **写真アプリ** | 蔵衛門 / 写真小町 | 1人880〜1,200円 | 工事写真の撮影・整理・台帳生成 |
| **AI橋梁点検** | ひびみっけ（富士フイルム） | 月数万円〜 | スマホ撮影+AIでひび割れ自動検出 |
| **Webカメラ・遠隔臨場** | Safie Pocket / iSPEC | 1台月5,000〜10,000円 | 段階確認・遠隔確認 |
| **電子契約** | クラウドサイン / GMOサイン | 1社月10,000円〜 | 契約書・請求書・発注書の電子化 |
| **AI損傷図作成** | 朝日航洋・NEC等のサービス | 1橋50〜150万円 | ドローン撮影＋AI解析の外注 |
| **GNSS/RTK測量** | ジェノバVRS等 | 月3〜15万円 | i-Con対応の測量 |

### 導入の段階イメージ

```
Step 1（即日〜1ヶ月）: 写真アプリ + チャット導入
  → 「写真整理が楽になった」「電話が減った」を体感

Step 2（1〜3ヶ月）: クラウドストレージ + 電子契約導入
  → 「書類を探さなくていい」「ハンコのために事務所に戻らなくていい」

Step 3（3〜6ヶ月）: AI橋梁点検 + 遠隔臨場
  → 「点検が半日で終わる」「自治体確認待ちが減る」

Step 4（6ヶ月〜）: GNSS測量・i-Con対応
  → 「ICT原則化に対応できる」「受注力が上がる」
```

## 4. CiviLinkが「横串」になる：ドキュメント管理とやり取りの一元化

ステップ3までで「点」の効率化はできます。しかし、各ツールで生まれたデータが**ツールごとにバラバラのまま**では、フェーズ間のデータ受け渡しの問題は解決しません。

ここで**CiviLink**が「横串」を通します。

### コンセプト

> **「ばらばらのツールを束ねて、図面と書類のやり取りをひとつの場所に」**

地場建設会社にとってCiviLinkがもたらす価値は、シンプルに3つです：

#### ① 探す時間をゼロにする
- 図面・書類・写真・指摘・承認履歴が**全て図面に紐付いて一元管理**
- 「あの図面どこ？」「あの指摘どうなった？」がワンクリックで分かる
- 過去の補修履歴・指摘履歴が検索可能

#### ② 承認待ちをなくす
- 自治体監督員と施工会社の間の照査・承認をCiviLink上でリアルタイム化
- 紙・FAX・メールの往復、押印待ちを排除
- 段階確認・材料承認・打合簿が画面上で完結

#### ③ 記録が勝手に残る
- 照査コメント・承認履歴・図面バージョンが**自動的に監査証跡として蓄積**
- 完成図書・電子納品のベースデータが施工中に自然と出来上がる
- 監査・引継ぎ・トラブル時の根拠資料に困らない

### 既存ツールとの関係（イメージ）

```
                    ┌─ チャット ─┐
                    │ (LINE WORKS) │
                    │             │
       写真アプリ ─┼─────CiviLink─────┼─ Webカメラ
       (蔵衛門等)   │  （図面起点で │  (Safie等)
                    │   横串連携） │
                    │             │
                    └─ クラウド ─┘
                       (Google等)
```

- 既存ツールで撮った写真・送ったメッセージ・確認した映像が、**全てCiviLink上の図面に紐付く**
- CiviLinkが各ツールのデータを束ねる「**ドキュメント管理基盤**」として機能
- 個別ツールの良さ（操作性・専門機能）はそのまま活かす

## 5. 「調査・設計・施工」DXマトリクス

CiviLinkと既存ツールが、各フェーズでどう機能するかの全体像：

| フェーズ | DX対象業務 | 既存ツール | **CiviLinkの役割** |
|---------|-----------|-----------|------------------|
| **① 調査（点検）** | 橋梁近接目視点検、損傷記録、点検調書作成 | ・ひびみっけ（AI損傷検出）<br>・ドローン外注<br>・蔵衛門（写真管理） | ・点検写真・損傷図を**橋梁図面に紐付け一元管理**<br>・現場タブレットで図面+過去履歴を即時閲覧<br>・損傷箇所を図面上にマーキング・コメント<br>・点検結果を自治体担当者と直接共有 |
| **② 設計（補修設計）** | 補修設計図面の照査、数量計算書との整合確認、設計変更管理 | ・CADツール<br>・クラウドストレージ（CAD原図） | ・**デジタル照査・指摘管理**（赤黄チェックの代替）<br>・図面バージョン管理・差分自動可視化<br>・設計変更の承認ワークフロー<br>・図面⇔数量計算書の整合チェック（将来AI機能） |
| **③ 施工（補修工事）** | 施工計画書、工事打合簿、段階確認、材料承認、出来形管理、工事写真、電子納品 | ・LINE WORKS（チャット）<br>・蔵衛門（工事写真）<br>・Safie（遠隔臨場）<br>・クラウドサイン（契約） | ・**工事打合簿のデジタル化**（図面上のコメント＋承認）<br>・段階確認の写真・図面紐付け<br>・材料承認ワークフロー<br>・出来形管理値の規格値チェック（将来AI）<br>・電子納品フォルダの自動構成<br>・完成図書の一元管理 |

### 横串の価値（フェーズ間のデータ受け渡し）

調査で記録した損傷データ → 設計の入力資料に
↓
設計で確定した補修図面 → 施工の作業基準に
↓
施工で記録した出来形・写真 → 次回点検時の前提情報に

このサイクルが**現状はExcel・メール・紙で分断**されています。CiviLinkが図面ベースで集約することで、**「データが流れる」維持管理サイクル**を実現します。

## 6. POC計画：佐賀市で何を実証するか

### POC期間
2026年5月〜2027年3月（包括委託開始までの約11ヶ月）

### 候補スコープ

| 項目 | 案 |
|------|-----|

### 段階的な実証スケジュール

```

2027年 1〜3月: 補修工事フェーズで打合簿・段階確認のデジタル化
2027年 4月〜: 包括委託開始、POC成果を全契約建設会社へ展開
```

| 領域 | 指標 | 目標 |

## 8. 将来展望：AI活用とデータ蓄積

POCで蓄積したデータをもとに、将来的にはAI機能でさらに業務を自動化します。

### Layer 2（AI業務代行）の将来展開

| 機能 | 概要 | 提供時期 |
|------|------|---------|
| AI損傷検出（CiviLink連携） | 点検写真からひび割れ・剥離をAI自動検出、損傷図自動生成 | 短期 |

マルメのエンジニアが現場に入り、**AIの定着・カスタマイズ・効果測定**を伴走支援します。地場建設会社にIT人材がいない問題を、外部リソースで補完します。

POC期間中に蓄積されるデータ（点検履歴・補修履歴・指摘履歴）は、**佐賀市の橋梁維持管理の貴重なナレッジ資産**になります。
