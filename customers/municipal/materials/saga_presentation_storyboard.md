## Slide 02: 事業の背景と目的
**レイアウト:** 3つの背景 + 目的ボックス

**背景:**
- **国の政策**: i-Construction 2.0により、令和8年度から舗装工・地盤改良工でICT施工が原則化。電子納品の対応範囲も拡大。対応できない企業は公共工事の受注が困難に
- **業界構造**: 建設業就業者の37%が55歳以上、29歳以下は12%。2035年には技術者約3.2万人が不足。佐賀市の公共工事を担う地場建設会社の技術基盤が揺らいでいる
- **制度環境**: 令和6年4月から建設業にも時間外労働の上限規制が適用。現場後の書類作成・写真整理等のアナログ業務が残業の温床であり、DXによる業務効率化が経営課題に直結

**目的:**
- 佐賀市の公共工事を担うCランク以下の地場建設会社に対し、ICT・AIツールの導入支援と図面データ基盤の整備を行い、インフラ維持管理体制の持続可能性を確保する

## Slide 03: 地場建設会社の現状と課題
**レイアウト:** 左右2カラム（現状の業務フロー / 課題の構造）

**現状の業務フロー（橋梁補修工事の例）:**
```
補修工事を受注 → 設計図面をメールで受領、社内サーバーに保管
  → 施工計画書をExcel/Wordで作成（過去の書類をコピー＆修正、1〜2週間）
    → 工事写真を数千枚撮影 → PCに手動転送 → フォルダで手動仕分け
      → 打合簿をメール添付で往復 → 修正のたびにバージョンが増殖
        → 段階確認は日程調整に時間がかかり、自治体担当者の現場立会を待つ
          → 完成図書・電子納品を工事完了後にまとめて作成 → 形式エラー・抜け漏れが多発
```

**構造的な課題:**
- **書類が散在**: 図面・写真・帳票がPC・社内サーバー・メールに分散。「最新版はどれか」「あの図面は○○さんしか知らない」が日常
- **やり取りが非効率**: 自治体⇔建設会社⇔設計者の連絡がメール・電話・対面に分散。添付ファイルのバージョン管理ができず、承認に数日〜数週間
- **データが流れない**: 点検データが設計に活きず、設計データが施工に活きない。毎フェーズで再入力や転記が発生
- **何から始めればいいか分からない**: 市場にDXツールは多数あるが、自社の規模・業務に合ったものを選ぶ知見がない

## Slide 04: 何もしないとどうなるか
**レイアウト:** 3段階のリスク連鎖図

```
【リスク①】地場建設会社がICT施工に対応できない
  令和8年度からICT施工原則化 → 対応できない企業は入札参加が困難に
  → 佐賀市の公共工事に参加できる地場企業が減少

      ↓

【リスク②】インフラ維持管理の担い手が不足する
  橋梁2,842橋の5年周期点検、900橋の自前点検、道路パトロール、
  補修工事の施工… これらを担う企業が減れば、維持管理サイクルが回らなくなる

      ↓

【リスク③】市民生活の安全に影響する
  点検の遅延 → 損傷の見落とし → 補修の後手
  長寿命化計画（50年で70億円のコスト縮減）の前提が崩れる
```

**補足:**
- 全国の市区町村の約4割が技術系職員5名以下。佐賀市も例外ではない
- ベテラン技術者の退職により「判断できる人」が減少。DXによるデータ蓄積・技術継承が急務

## Slide 07: 取組②: 図面データ基盤の整備
**レイアウト:** Before/After + 3者連携モデル図

**Before（現状）:**
```
自治体 ──メール/USB──→ 設計会社 ──メール/USB──→ 建設会社
  ↑                                                    │
  └──────── FAX/紙/メール ← 打合簿・段階確認 ─────────┘
  
  問題: 誰がどの版の図面を見ているか分からない
        承認に数日〜数週間。「言った言わない」が日常
        完成図書は工事後にまとめて作成 → エラー多発
```

**After（データ基盤整備後）:**
```
               ┌──────────────────────────┐
               │  クラウド図面データ基盤     │
               │                          │
  自治体 ─────┤  点検データ → 設計データ    │───── 設計会社
               │      ↑                   │
  建設会社 ───┤  施工データ → 完成図書     │
               │                          │
               │  全員が同じ図面を見る      │
               │  承認・照査はクラウド上で即時│
               │  記録が自動的に蓄積される   │
               └──────────────────────────┘
```

**3者にとっての効果:**
- **自治体（道路整備課等）**: 監督員の照査・承認業務を50%削減。技術継承のデータ基盤
- **補修設計会社**: 照査やり取り日数を30%削減。電子納品準備を70%削減
- **地場建設会社**: 打合簿往復時間を70%削減。残業を月20時間削減

## Slide 08: 取組③: DX伴走支援
**レイアウト:** 伴走支援の内容 + なぜ必要か

**なぜ伴走支援が必要か:**
- ツールを導入しても「現場の業務に合わない」「結局使われなくなる」で終わるケースが全国で多発
- 土木の現場を理解した人間が、現場に入って一緒に使いながら業務フローごと改善する必要がある
- 業務改善の知見がないままツールだけ入れても効果は出ない

**伴走支援の内容:**

| フェーズ | 内容 | 期間 |
|---|---|---|
| **導入準備** | 対象企業の現状IT環境調査、業務フロー分析、最適ツール選定、補助金申請支援 | 1ヶ月 |
| **導入・研修** | ツールの初期設定、操作研修（現場＋事務所）、マニュアル作成 | 1〜2ヶ月 |
| **定着支援** | 週1回の現場訪問、困りごと対応、業務フローの改善提案 | 3〜6ヶ月 |
| **効果測定** | 導入前後の工数比較、コスト削減額の算出、改善点の特定 | 随時 |

**実施体制:**
- 伊藤忠商事のDXパートナー企業（土木技術者約20名＋IT技術者を擁する）が実施
- 土木とITの両方を理解する専門人材が現場に入る
