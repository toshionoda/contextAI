# 佐賀市向け 統合戦略：技術ランドスケープ × CiviLinkメニュー × 3者連携

**作成日:** 2026-04-14
**目的:** 市場に存在する土木AI/DX技術（52件）とCiviLinkサービスメニュー（24件）を統合整理し、佐賀市POCで「何を」「誰に」「どの順番で」提案するかを明確にする
**根拠資料:** 技術カタログスプレッドシート、CiviLinkサービスメニュー、3者連携メニュー、DX支援メニュー

---

## 1. 全体構造：3つのレイヤーで整理する

```
┌──────────────────────────────────────────────────────────┐
│  Layer A: 市場の技術ランドスケープ（52件）                   │
│  → 「世の中にはこれだけの選択肢がある」を見せる武器           │
├──────────────────────────────────────────────────────────┤
│  Layer B: CiviLinkサービスメニュー（24件）                   │
│  → マルメが提供できるもの。Phase×Layer×適用性で優先順位化     │
├──────────────────────────────────────────────────────────┤
│  Layer C: 佐賀市3者連携メニュー（A1〜A7, B1〜B7, C1〜C8）    │
│  → 佐賀市の3者（自治体・補修設計・地場建設）に刺さる具体提案  │
└──────────────────────────────────────────────────────────┘
```

**この3レイヤーの関係:**
- Layer A（市場）を知っているからこそ「なぜCiviLinkか」を語れる
- Layer B（自社メニュー）の中で「佐賀に刺さるもの」がLayer C
- Layer Cは4/20訪問の提案コア。Layer A・Bは裏付けと将来展望

## 3. CiviLinkサービスメニュー：適用性による優先順位

スプレッドシートの「適用性」評価をもとに、佐賀市POCでの優先度を4段階に分類。

### Tier 1: 佐賀市に「刺さる」メニュー（適用性: あり）

| # | メニュー | Phase | Layer | 適用性の理由 | 3者連携での位置 |
|---|---------|-------|-------|------------|---------------|
| 12 | **図面⇔数量計算書の整合性チェック** | Phase 3 | Layer 2 | 補修設計の照査で最大の手戻り要因。自治体・設計会社双方に価値 | B4（補修設計会社向け）、A3（自治体向け） |
| 14 | **施工計画書の自動照査** | Phase 3 | Layer 2 | 自治体監督員の最大負担業務。必須記載事項のチェック自動化 | A2（自治体向け仕様書ドラフト） |
| 15 | **積算ミス検知** | Phase 3 | Layer 2 | 過去データとの比較で桁違い等を自動検出。自治体の予算管理に直結 | A2、B4の発展形 |
| 21 | **紙図面のデジタル化支援** | Phase 1 | Layer 3 | 佐賀市の昭和〜平成期の紙図面が大量に存在。CiviLink活用の前提条件 | C1（地場建設会社）の前段階 |

| # | メニュー | Phase | Layer | 適用性コメント | 3者連携での位置 |

### スライド3「世の中の選択肢」の見せ方

技術カタログ52件をそのまま見せるのではなく、以下の切り口で**「マルメは市場を熟知している」ことを示す**:

```
「土木AI/DXは52以上の製品・技術が市場に存在します。
 しかし、地場建設会社が"今すぐ使えて効果が出る"ものは限られます。」

  ◎ 地場適性が高い（即導入可能）     ─── 18件
  ○ 導入可能だが環境整備が必要       ─── 24件
  △ 大規模or研究段階                ─── 10件

「マルメは、◎の中から佐賀市の橋梁維持管理に最適な組合せを提案します。」
```
