## Part 1: 診断 — 現在地

### 1.1 起業の科学フレームでの現在地

**PSF（Problem-Solution Fit）部分達成・PMF未達・死の谷で停滞**

```
[CPF]      [PSF]            [PMF]              [SMF/Scale]
課題発見  → 解決策フィット → プロダクト市場  → スケール
                            フィット
                            
✅完了    ⚠️非対称達成     ❌死の谷停滞      ❌時期尚早
```

### 1.2 各フェーズの判定

#### CPF：✅達成
- 「照査が時間かかる」「指摘管理が散らかる」「DocuWorksが使いづらい」というペインは確認済
- Job-to-be-Doneは把握できている

#### PSF：⚠️非対称達成

| 対象 | PSF状態 | 証拠 |
|---|---|---|
| Early Adopter（栗山さん的） | ✅達成 | 栗山さんが投資家に推薦／渋谷さんIssue週92回／JFE 4名本気利用 |
| Early Majority（5人の道路設計者・三井住友16名・JFE 32名等） | ❌未達 | 「コンセプトは良い」（拒絶）／触らずに離脱／廃止宣言ゼロ |

**Early Adopter市場ではPSF達成、Early Majority市場ではPSF未達**。これが「死の谷」の典型的な姿。

#### PMF：❌未達

| 項目 | 達成度 |
|---|---|
| Sean Ellis Test 40%以上「とても残念」 | おそらく未達 |
| オーガニック成長 | ❌未発生 |
| Cohort Retention 高い | ❌（1ヶ月後も触らない人多数） |
| 「ないと困る」発言 | 栗山さん級のEarly Adopterのみ |
| CAC < LTV | ⚠️疑わしい（30万円ARPUでCAC高） |
| 紹介・口コミ普及 | ❌（栗山さん経由の福山1社程度） |

#### SMF：❌時期尚早

PSFも部分的、PMFも未達なのに、営業の行動様式がSMF的（全国デモ・件数追跡）になっている。**フェーズと行動のミスマッチ**が構造的問題。

### 1.3 死の谷で詰まる典型パターン全該当

| 典型パターン | CiviLink該当 |
|---|---|
| VoCを聞きすぎて機能が肥大化 | ✅ |
| トライアル提供を続けるがCVR上がらない | ✅ |
| 「コンセプトは良い」「あったら便利」止まり | ✅ |
| 製品を作り続けるが PMF達成しない | ✅ |
| Early Adopter偏重で Early Majority と乖離 | ✅ |
| ペルソナを絞り込めず全方位で薄く戦う | ✅ |
| Aha! Moment が再現性ない | ✅ |
| Loss Interview を実施していない | ✅ |

→ ほぼフルセットで該当。

### 1.4 ボトルネック分析

```
リード →[A]→ デモ →[B]→ 有償化 →[C]→ 定着（WF乗っ取り）→[D]→ キャズム越え
       過剰      漏れる       付き合い的       ほぼゼロ
```

**根本ボトルネック：[C] ワークフロー乗っ取りを完遂した参照事例ゼロ**

有償16社を抱えていながら、「DocuWorks廃止宣言」したような顧客が事実上ゼロ。福山30万/6ヶ月もこの典型で、有償だが定着してない。

#### 優先順位（解決の順序）

1. 🥇 **有償化→定着（[C]＝ワークフロー乗っ取り完遂）** — レバー最大、なくては他が動かない
2. 🥈 **営業戦略（3社集中・KPI再設計）** — [C]の成功確率を上げる前提条件
3. 🥉 **デモ（業務診断セッション化）** — 戦術レベル、[C]成功後に最速で効く
4. **キャズム越え** — 12-24ヶ月先、今手をつける段階ではない

## Part 2: 戦略フレーム — 理論基盤

### 2.1 キャズム理論の正しい理解

**キャズムは「マーケティング問題」ではなく「信頼問題」**

早期多数派（プラグマティスト）が買う唯一の理由：
> **「自分と同じ業界・同じ規模・同じ業務の会社が使っていて、成果を出している」**

機能・価格・数字・VoC・ROI試算は全部副次的。これ以外はノイズ。

| 指標 | キャズム手前 | 超えた |
|---|---|---|
| 顧客獲得 | こちらから営業 | 同じセグメントから問い合わせが来る |
| 業界内認知 | 「聞いたことある」 | 「あそこも使ってる」が会話に出る |
| ユーザー側 | こちらが運用支援 | ユーザー同士の勉強会が自然発生 |
| 案件サイクル | PoC→検討→稟議で6ヶ月 | 「他社使ってるからうちも」で2ヶ月 |

### 2.2 「超えた」の定義

**ニッチ内浸透50%以上の状態**。

- 福山1社では超えられない（n=1は社内提案が通らない）
- 大手の事例も中堅には届かない（規模が違う＝「うちには関係ない」）
- 業界誌に出ても「ベンダーが書かせた」と判定される

### 2.3 MVPの二重性

**Early Adopter MVP（Vision MVP）とEarly Majority MVP（Whole Product MVP）は別物**

| | Vision MVP | Whole Product MVP |
|---|---|---|
| 対象 | 栗山さん的なイノベーター | 渋谷さん的な現場担当者 |
| 買う理由 | 業界の未来を見ているから | 月曜日の業務が回るから |
| 最低条件 | コア差別化機能＋ビジョン | 業務の100%代替 |
| ギャップへの態度 | 自分で埋める／待てる | 埋まってないなら使わない |
| 不完全さ | 「面白い、これから良くなる」 | 「未完成」＝拒絶 |
| 必要な周辺 | 少ない | 膨大（教育・移行・サポート・連携） |

**「目が死んだ」現象の正体**：Vision MVPは栗山さんに刺さったが、Whole Product MVPが未成立で5人の道路設計者には「未完成」に見えた。

### 2.4 ワークフロー乗っ取り — 普及メカニズム

ライセンス販売の積み上げで広がるSaaSはほぼない。実際に広がるのは **「業務上、CiviLinkを開かないと仕事が回らない人」が増える** 構造。

**ANDPADアナロジー**：「現場監督が現場写真をANDPADに上げる」と決めた瞬間、職人・施主・本社の全員がANDPADを開かないと写真が見られない。**強制力 → 必須ユーザー化**。

#### 福山コンサルでのスケール段階

```
現在: 栗山(東京) ── 渋谷(北九州)        ← 19名中2名、便利ツール状態
                       ↓ 渋谷が照査WFを CL に一本化（DW廃止）
第1段: 渋谷 + 照査対象の設計者 5-10人    ← 毎日触る依存状態
                       ↓ 部門長が部門標準と宣言
第2段: 道路設計部門 30-50人              ← 全員ライセンス必須
                       ↓ 他部門の照査担当が真似
第3段: 橋梁・河川・地盤の照査部門         ← 全社展開
```

各段の遷移は**人数で起きるのではなく、決定で起きる**：
- 第1段：渋谷さんの「DocuWorks廃止宣言」
- 第2段：部門長の「部門標準宣言」
- 第3段：役員の「全社標準宣言」

## Part 3: ペルソナと訴求の再定義 — Loss Interview結果

### 3.1 Loss Interviewからの決定的データ

野田さんが聞いた2つの発言：

1. **「チームで使うイメージがわかない（少人数で業務を実施するため）」**
2. **「図面チェックを適切にするほど図面枚数多い業務ではないから、イイのはわかったけどタイミングがない」**

### 3.2 これが示すもの

**CiviLinkの主訴求軸（チーム協業・枚数効率化）が業界の半分以上に刺さらない**。

#### 業界実態
- 道路設計：1案件1〜2人体制が標準
- 橋梁詳細：設計者1＋照査担当1
- 都市計画：個人主体
- 「3〜5人のチームで協業」は一部大型案件に限る

#### 図面の実態（4/28福山ブリーフィングと一致）
> 「道路設計の図面は枚数が少ないけれど、1枚あたりの密度が異常に高い仕事」

道路設計だけの話ではなく、業界全体で「枚数が多い業務」の方が少数派。

### 3.3 訴求軸の組み直し

| | 旧訴求（多くの場面で滑る） | 新訴求（業界実態に合致） |
|---|---|---|
| 対象規模 | 3〜5人チーム | **1〜2人の照査担当者** |
| 業務量 | 図面枚数多い案件 | **図面1枚あたりの密度高い案件** |
| 価値の場所 | 効率化・自動化 | **深い照査・履歴参照・協議透明化** |
| 時間の使い方 | 「3時間→30分」（短縮訴求） | 「**5年前の指摘を5分で参照**」（質訴求） |
| 協業の単位 | チーム内協業 | **発注者・外注先との時系列協業** |
| キラーフレーズ | 「チームの生産性向上」 | 「**1枚の図面の30年分の文脈を全部見える化**」 |

### 3.4 渋谷さんの行動データとの整合性

- 渋谷さん：Issue週92回 ＝ **個人で大量に使っている**（チーム協業ではない）
- **個人ヘビーユーザー**として Aha! Moment に到達済み

→ 渋谷さんのアハは「チーム機能のアハ」ではなく「個人深堀り機能のアハ」。これがピン1集中の真の意味。

### 3.5 ピン1 Whole Product MVPの修正された構成要素

```
[コア機能]                  ← 既存（強み）
- Issue（指摘管理）         ← 渋谷さん週92回利用＝Aha到達
- 図面差分比較
- 図面表示

[個人ユーザー深堀り]        ← 強化必要
- 過去案件の指摘の座標検索
- 1枚の図面への大量指摘の効率的記入
- 基準・要領との照合補助
- 外注成果品の照査ワークフロー

[履歴・協議]                ← 新カテゴリ・不足
- 協議の往復履歴の時系列表示
- 「いつ誰がどう判断したか」の追跡
- A案/B案/C案の比較管理
- 路線連続性（座標ベース）

[完全代替性]                ← 不足
- 印刷・PDF出力の安定動作
- 変換完了通知
- DocuWorks資産の取り込み
- 周辺機能（検索・スタンプ等）

[ワークフロー適合]          ← 福山特有
- 詳細設計成果品の照査テンプレ
- 福山社内のフォルダ命名規則対応
- 標準納品形式への対応

[業界基準]                  ← 道路設計特有
- 道路構造令・設計要領との連携
- 整備局基準の指摘テンプレ

[教育・移行]                ← ほぼ無
- オンボーディング動画
- 「DocuWorksでこう→CiviLinkでこう」マップ
- 部門長向け説明資料
- 設計者向け15分ハンズオン

[稟議・信頼]                ← 部分的
- 投資対効果計算（栗山さん自身が測れるツール）
- 段階的導入プラン
- セキュリティ説明資料
- 競合比較表

[弱体化対象]                ← 削除候補
- チーム協業前提の機能
- 「3-5人のチーム」前提のセットアップ
- 大量図面捌き前提の効率化UI

[スコープ外（DocuWorks併用継続）] ← Pin2以降または別ピンに温存
- 数量計算書のバインド・表紙/目次作成
- 成果品取りまとめ・束ねUX
- 納品用PDF生成・ページ並べ替えの汎用ツール
```

→ 詳細は §3.6.1「DocuWorks廃止宣言」の用語定義を参照。

### 3.6 判定基準（Whole Product MVPか否かの唯一の問い）

> **渋谷さんが「今後、私の照査指摘は CiviLink にしか入れません」と部門内に宣言できる状態か？**

機能数でも契約金額でも利用率でもなく、**この行動が起きるかどうか**で判定する。

#### 3.6.1 「DocuWorks廃止宣言」の用語定義（スコープ境界）

本書で「DocuWorks廃止宣言」と呼ぶのは、**照査ワークフローにおける指摘の置き場としての廃止**であって、DocuWorksの全用途廃止ではない。実態は「**照査指摘の唯一の置き場宣言**」。

| 区分 | 対象 | CiviLinkで代替するか |
|---|---|---|
| ✅ ピン1のスコープ内 | 照査指摘の記入・参照・履歴管理 | する（必須） |
| ✅ ピン1のスコープ内 | 図面表示・差分比較・座標検索 | する（必須） |
| ✅ ピン1のスコープ内 | 印刷物との整合性・DocuWorks資産取り込み | する（廃止宣言の阻害要因解消として必須） |
| ⛔ ピン1のスコープ外 | 数量計算書のバインド・表紙/目次作成 | **しない**（DocuWorks併用継続を許容） |
| ⛔ ピン1のスコープ外 | 成果品取りまとめ・束ねUX | **しない**（DocuWorks併用継続を許容） |
| ⛔ ピン1のスコープ外 | 納品用PDF生成・ページ並べ替えの汎用ツール | **しない**（DocuWorks/Acrobat等で代替可） |

**理由**：照査指摘UXと成果品取りまとめUXは別ジョブ。前者はCiviLinkの強みで攻める領域、後者はDocuWorksが成熟しており短期で代替する戦略的意義が薄い。PSF段階の集中戦略を守るため、後者はPin2以降または別ピンに温存する。

**運用上の含意**：渋谷さんは照査指摘をCLに一本化した後も、成果品まとめ時にはDocuWorksを開く前提。これは廃止宣言の阻害要因ではない（前述の通り「廃止宣言」=照査指摘の置き場の一本化、を意味するため）。

## Part 4: 営業・購買モデルの再設計

### 4.1 「無料トライアル→有償化」モデルの放棄

無料トライアルが機能する4条件をCiviLinkは**1つも満たしていない**：

| 条件 | CiviLinkの実態 |
|---|---|
| セルフサーブで価値が伝わる | ❌ |
| Time to Value が数日以内 | ❌（数ヶ月） |
| 個人または小チームで決裁できる価格帯 | ❌（法人決裁・稟議必須） |
| 既存ワークフローを邪魔せず追加で入る | ❌（DocuWorksを置き換える必要） |

→ モデルそのものを捨てる判断が要る。

### 4.2 3段階購買プロセス

```
[Stage 1] 業務診断セッション（半日〜1日・対面）
   目的: ワークフロー観察＋仮説提示＋次の相手見極め
   提供物: デモアカウント24-48h（その場で発行・期限後消滅）
   売り込み禁止
                ↓ 3条件揃った場合のみ
[Stage 2] 有償プロトタイプ（¥1.5M〜¥3M / 2-3ヶ月）
   目的: 「DocuWorks廃止宣言」をしてくれる5-10人チームでワークフロー乗っ取り完遂
   提供物: ライセンス＋CS週次伴走＋効果測定支援＋社内稟議用レポート納品
   並走禁止条項
                ↓ プロトタイプ成功時
[Stage 3] 年契約（¥6M〜¥30M）
   部門標準導入・部門長スポンサー・3年契約前提
```

**無料トライアルというステージは存在しない**。デモか有償プロトタイプの二択。

### 4.3 なぜ¥3M枠が効くか

| 30万/6ヶ月（旧） | 300万/3ヶ月（新） |
|---|---|
| 月5万：「付き合いで」が通る | 月100万：「成果出さないと社内ヤバい」 |
| CSが「お忙しいところ申し訳ない」 | CSが「毎週VOCします（料金分の責務）」 |
| 失注時もLoss Interview打診しづらい | 失注時もヒアリング当然の前提 |
| 並走を許容してしまう | 並走したら投資対効果がゼロなので顧客側も止める |

**故意に高くする**のが戦略。安いと真剣になってもらえない。

### 4.4 デモ → 業務診断セッション

| | 旧デモ | 新「業務診断セッション」 |
|---|---|---|
| 誰のため | 売り手のため（クロージング装置） | 双方のため（フィット判断装置） |
| やること | 機能紹介・スライド | 業務観察・ペイン抽出・仮説ぶつけ |
| しゃべる比率 | 売り手70%・客30% | 売り手30%・客70% |
| 次に繋ぐ先 | 無料トライアル | 有償プロトタイプ or クリーンに離脱 |
| 成功判定 | 「コンセプトは良いですね」 | 「うちの◯◯業務でやりませんか」 |

#### 業務診断セッションの3つのアウトカム

1. **A：双方が次に進む合意** → プロトタイプ提案書を1週間後に送付

## Part 5: 顧客理解の罠と対処

### 5.1 Early Adopter偏重ヒアリングの罠

#### バイアスのメカニズム

Early Adopterは**自分で工夫してギャップを埋める**ので、不足を感じる頻度が低い。「あれば便利」レベルのVoCしか出ず、Early Majorityの決定的不足は出てこない。

#### 構造的偏り

```
営業の入口   → コネを持つ人＝先進性ある人＝Early Adopter
時間取れる人 → 興味ある人＝Early Adopter
紹介の連鎖   → Early Adopter経由で繋がる人＝Early Adopter
快くインタビューに応じる → Early Adopter
失注した相手 → 「もう関わりたくない」で逃げる ← Early Majorityが多い
```

→ 意図的に対処しない限り、ヒアリングは必ずEarly Adopterに偏る。

#### 偏りに無自覚だと起きること

- 「現場のVoCを聞きました」と言える（誠実さの自覚あり）
- でも実態は **Early Adopterの声しか聞いてない** → Early Majority向けに合わない製品が出来る
- 「ヒアリングしたのに普及しない」という二重苦が発生

### 5.2 栗山さん vs 照査部門の人 — 役割分担

#### 栗山さんで判断できない3つの理由

1. **役割が違う**：栗山さんは東京支社・管理上流。実際に照査をやる人ではない
2. **見えている景色が違う**：「業界の未来像」vs「明日の業務」
3. **判定基準が「渋谷さんの行動」だから**：栗山さんの評価ではMVP判定できない

#### 役割の分担

| 栗山さんに聞く（活かす） | 照査部門の人（渋谷さん）に聞く（活かす） |
|---|---|
| 業界・福山の経営戦略 | 1日の照査業務の時系列観察 |
| 経営層・上司の関心事 | DocuWorksで具体的に何をしているか |
| 部門間の力学・人脈 | CiviLinkに乗り換える時の阻害要因 |
| **照査部門の人の逆紹介** | 「DocuWorks廃止宣言」できない理由 |
| 戦略的判断・プロトタイプ組み立て | 印刷・出力・資産取り込みの具体ペイン |

#### 罠

「栗山さんに聞けばわかる」は **「窓口が信頼できる人だから情報も信頼できる」** という誤認。栗山さんへの信頼と、栗山さんから得られる情報の妥当性は別の話。

### 5.3 観察 > インタビュー

Early Majorityは時間取らない・本音言わない。だから：
- **観察**：オフィス訪問して横で業務を見る（インタビューより情報量多い）
- **行動データ**：Mixpanel等で使用ログ、離脱パターン分析
- **Loss Interview**：離脱者が本音を出すタイミング
- **「使ってない人」へのアプローチ**：JFE 32名・三井住友16名等

### 5.4 Loss Interviewの正式化

#### 対象
- 四国建設コンサル
- フクシコンサル
- 日本シビック横尾様
- 三井住友建設鉄構エンジ
- JFE設計（4名アクティブ以外）

#### プロセス
- フロント営業以外（CS担当）が30分電話/対面でヒアリング
- 5項目テンプレ：誰が／いつ／何と比較して／何が足りなくて／いくらなら通ったか
- 月次レポート化

#### 「枚数」「チーム」発言を必ず深掘り

野田さんの聞いた2発言（チーム使用イメージ無し・枚数訴求ズレ）が他のLoss Interview対象でも出るか確認。出れば訴求軸の修正が確定。

### 5.5 栗山さん経由の「逆紹介」

```
「栗山さんと違うタイプの人を紹介してください。
CiviLinkに前向きじゃない、技術にあまり興味がない、
忙しくて新しいツール触らない、でも照査の実務をやっている人」
```

Early Adopterは「自分と違うタイプ」を社内で把握している。最速のEarly Majorityアクセスチャネル。

## Part 6: 組織設計 — CS主導 Discovery

### 6.1 役割の再分割：What と How

```
[Discovery：何を作るか]      → CS主導
- Customer Journey
- Whole Product 不足リスト
- ユーザーストーリー
- 機能仕様（PRD）
- アクセプタンスクライテリア
- 優先順位

[Delivery：どう作るか]       → プロダクト主導
- 技術選定
- 工数見積もり
- アーキテクチャ判断
- 実装
- 技術的トレードオフ提示

[Validation：使えるか]       → CS主導 + プロダクト協力
- 福山プロトタイプでの検証
- 「DocuWorks廃止宣言」獲得
- PSF達成判定
```

CSは「これを作れ」と命令するのではなく、**「これを解決したい」をPRDで渡す**。解決方法はプロダクトと共同設計。

### 6.2 CS主導が正しい理由

| 観点 | プロダクト主導 | CS主導 |
|---|---|---|
| 顧客接点 | 間接（営業伝言） | 直接（Journey観察） |
| Early Majority視点 | Early Adopter偏重に流れがち | 顧客接点が幅広いため補正可 |
| Build Trap | 起きやすい | 「使われる機能」起点で回避 |
| フィードバックループ | 営業→PdM→開発で長い | CS→開発で短い |
| PSF達成への直結性 | 弱い | 強い |

### 6.3 CSの標準アウトプット

#### Customer Journey フォーマット
- ペルソナ（固有名詞ベース）
- 時系列ステップ
- 各ステップの使用ツール
- 摩擦点・離脱点
- 感情曲線

#### Whole Product 不足リスト
- 不足項目
- どの Journey ステップで起きるか
- 該当ペルソナ
- 「DocuWorks廃止宣言の阻害要因解消か」の判定
- 観測利用データ（Mixpanel等）

#### ユーザーストーリー
```
As a [渋谷さん的な照査担当者]
I want to [自分の指摘を CiviLink で記入する]
So that [照査指摘の置き場を CiviLink に一本化できる]
       （= §3.6.1 で定義する「DocuWorks廃止宣言」）

Acceptance Criteria:
- [ ] 印刷物との整合性確認ができる
- [ ] 過去案件の指摘を座標検索できる
- [ ] DocuWorks資産を一括取り込みできる

Out of Scope（Pin1では作らない・DocuWorks併用許容）:
- 数量計算書のバインド・表紙/目次作成
- 成果品取りまとめ・束ねUX
- 納品用PDF生成・ページ並べ替えの汎用ツール
```

### 6.4 CS主導の落とし穴と回避策

| 落とし穴 | 回避策 |
|---|---|
| ❌「○○さんがこう言ってた」を機能要件化 | 必ず Customer Journey 上の摩擦点として根拠化 |
| ❌ Early Adopter偏重の機能要望が紛れ込む | 「DocuWorks廃止宣言の阻害要因解消か」の単一判定軸 |

## Part 7: Customer Journey 主導の不足診断

### 7.1 なぜ Journey が機能整理より先か

| | 機能整理単独 | Customer Journey 先 |
|---|---|---|
| 視点 | Inside-out（自社視点） | Outside-in（顧客視点） |
| 問い | 「何を持っているか」 | 「彼らは何を経験しているか」 |
| 答えるもの | 機能の在庫・分類 | 摩擦・離脱・切替の瞬間 |
| 不足の発見 | 見えない（ない物の名前が分からない） | 直接的に出る |

### 7.2 作るべき3つのJourney

#### Journey A：渋谷さんの照査業務フロー（Workflow Journey）

1案件の照査が始まってから返すまでの時系列。各ステップで「どのツール使ってるか」「何に困ってるか」「なぜCiviLinkに完全切替しないか」を埋める。

#### Journey B：DocuWorks廃止までの心理ハードル（Switching Journey）

```
認知 → 試用 → 選好 → 決断 → 宣言 → 実行 → 維持

「決断が起きていない理由」が Whole Product 不足の核心
```

#### Journey C：4/28道路設計5人の感情曲線（Buying Journey）

「目が死んだ瞬間」の特定 ＝ Early Majorityが拒絶した具体的トリガーの特定。

### 7.3 Journeyの作り方（注意点）

1. **想像で書かない**：渋谷さん本人に1日横で観察させてもらう
2. **1ペルソナ1Journey**：固有名詞ベース
3. **感情曲線を必ず入れる**：離脱トリガーと決断トリガーの両方を捉える
4. **CSパネル含めて全タッチポイントを並べる**：摩擦は意外と非製品部分にある

## Part 8: 別チーム提案へのフィードバック

### 8.1 別チームのAテスト提案

```
1. 単純自動化（照査AI）
2. LLMでワークフロー解決 ← こちら
   - 若手技術者ペルソナ
```

「Aテスト：紙図面OCRによる指摘化でMVPになり得るか」

### 8.2 フレーミングの問題

#### ズレ1：Aテストの中身は実は不足診断

書かれている作業（離脱要因のヒアリング・ギャップ可視化）はABテストではなく**Whole Product MVPの不足診断**。命名と中身がズレている。

#### ズレ2：二択の立て方が戦略レベルで間違い

「単純自動化 vs LLMで解決」は同じレイヤー（機能追加で何を作るか）の議論。**戦略メモが言う「機能追加か営業戦略か」では「営業戦略が先」**。先に二択を立てるのが順番違い。

#### ズレ3：ペルソナ問題

「若手技術者」は**使う人だが買う人ではない**。早期多数派の購買判断は「照査担当者＋部門長＋稟議」の組み合わせ。

#### ズレ4：MVPはヒアリングで決まらない

「OCRがあればMVPですか？」と聞いても**仮説でしか答えてもらえない**。仮説回答と実際の購買行動は乖離する。

### 8.3 統合提案

```
旧：『紙図面OCRによる指摘化でMVPになり得るか』のAテスト

↓

新：『福山ピン1のWhole Product MVP不足診断』
   - 4/28訪問の道路設計5人ヒアリング深掘り
   - 渋谷さんの照査ワークフローを完全棚卸し
   - 「DocuWorks廃止宣言」の阻害要因を全列挙
   - 阻害要因を実装インパクト×実装コストで優先順位付け
   - 結果：OCRが上位かもしれないし、印刷バグ修正の方が上位かもしれない
   - 判断軸：高度機能の魅力ではなく『DocuWorks廃止宣言を可能にするか』
```

### 8.4 別チームへの伝え方

「LLM方向性は止めろ」と言うと反発される。むしろ：

> 「方向性は良い。OCRもLLMも有力候補。ただし**先に決め打ちする前に、ピン1（福山）で不足全体を一度棚卸ししませんか**。OCRが本当に最大ボトルネックか、それとも別のものが上位に来るかを観察で確かめる」

## Part 14: 価格モデル再設計 — 戦略議論で見落としていた重要論点

### 14.1 芙蓉コンサル土井内VoC（2025-12-19）で判明した価格問題

> 「**指摘事項一覧出力ができるとありがたい。指摘管理の一覧出力はメンバーでしたいが、現在の1万円という価格は高い**」
> 「**差分比較なしでレポート出力が可能な中間的な価格帯（現在の1万円と4000円の中間）のメンバー機能**の必要性」
> 「**ドメイン単位での契約（オーナー1人と担当3〜4人で月2万円程度）**が費用対効果の観点から望ましい」
> 「**ストレージ容量超過による追加料金が発生する場合、運用中に急な金額増を避けたい**」

### 14.2 現行価格モデルの構造的欠陥

| 現行 | 問題 |
|---|---|
| メンバー1万円 | 査読者中心の照査担当者には高すぎ |
| エンジニア4,000円 | 出力機能なし → 査読者が使えない |
| **中間価格帯がない** | 「指摘出力できるが差分比較なし」の現場ニーズに対応できない |
| ドメイン単位パッケージなし | 「オーナー1人＋3-4人で月2万円」の希望に応えられない |
| ストレージ追加料金変動 | 運用中の予算ブレが嫌われる（特に建設業の単年度予算） |

### 14.3 提案する価格モデル

```
[既存]
オーナー        : 通常価格
エンジニア      : 4,000円（差分比較・指摘記入のみ）
メンバー        : 10,000円（+ 指摘一覧出力）
サポートエンジニア : ?

[追加提案]
🆕 査読者プラン  : 7,000円（指摘記入＋一覧出力、差分比較なし）
🆕 ドメインパッケージ: 月20,000円（オーナー1人＋担当3-4人セット）
🆕 ストレージ定額化: 上限なしの月額固定オプション
```

### 14.4 有償プロトタイプ¥3M設計時の参考

戦略書Part 4の「有償プロトタイプ¥1.5M〜¥3M / 2-3ヶ月」設計時に：
- 査読者プランを含めた構成
- ドメインパッケージで「絶対成功スコープ」の5-10人分を一括提供
- ストレージ定額化込み

これで**並走禁止条項**と**価格設計の明確性**が両立する。

### 14.5 既存有償16社の価格モデル不適合の可能性

## Part 18: 若手技術者ペルソナの議論と却下経緯（**未確定 / 2026-05-09〜10討議**）

> ⚠️ **このPartは未確定**。野田さんとの討議メモを構造化したもので、戦略書本体への正式統合（v1.2化）は経営層合意後。

### 18.1 議論の発端

> 「営業マン複数いるので、ペルソナと戦略を複数作りたい。照査技術者・若手技術者でそれぞれキャズム突破したい」（野田さん 2026-05-09）

戦略書v1.1まではピン1=照査技術者（渋谷さん）に集中。営業リソース活用の観点から並列ペルソナ設計の必要性が議論された。

### 18.2 若手技術者ペルソナの構造的問題

照査技術者（渋谷さん型）はワークフロー乗っ取り（権威ある決定者の存在）が成立するが、**若手技術者は決裁権がないため、同じメカニズムが使えない**。

検討した3経路：

| 経路 | 内容 | 評価 |
|---|---|---|
| 経路1：照査ピンの配下ユーザー | 渋谷さん廃止宣言→部下若手5-10人が必須化 | 既存戦略の中・新ペルソナ不要 |
| 経路2：若手ニッチ商品化 | 新人研修・OJT補完ツールとして再パッケージ・Buyerは人事/教育担当 | 後述で却下 |
| 経路3：若手チャンピオン→上司ボトムアップ | 建設業界（年功序列）では弱い | 罠・採用しない |

### 18.3 若手線の機能ギャップ発見：外注管理ハブ型

経路2を進める前に「**現在の機能では価値が出ない**」（野田さん）と判明。具体的な不足機能の特定で **若手の業務の核は「外注管理」** と判明：

> 「**協力会社への指示や、修正されてきた図面をすぐに確認するときに、即時価値化するもの**」（野田さん）

これにより若手ペルソナは「OJT補完」ではなく「**外注管理ハブ型**」と再定義された：

| 項目 | 旧仮説（OJT補完） | 新仮説（外注管理ハブ） |
|---|---|---|
| 業務の核 | 学習・質問・知識習得 | 協力会社への指示出し＋成果物確認 |
| 立場 | 学ぶ側 | 社内では若手・対協力会社では発注者 |
| Buyer | 人事/教育担当 | 部長・課長（外注管理責任者） |
| 廃止宣言相当 | 新人研修にCL組み込み | 「外注成果物確認をCL標準化」決定 |
| 訴求 | 早期戦力化・知識継承 | 外注管理の生産性・指示伝達精度・差分確認の時短 |
| 価値検証 | 蓄積で効く | **即時価値化（1回目で体感）が必須** |

#### 戦略的含意：社外波及エンジン

「若手が外注先に出すフォーマット=CL」と決まった瞬間、協力会社も触らざるを得ない＝**ANDPADモデルが社内に閉じず社外（協力会社）へ伝播**。協力会社は複数の発注元を持つ→他建設コンサルへ自然伝播。照査線（社内乗っ取り）にない**横展開エンジン**になり得る。

#### 必要機能（3点）

1. **指示作成**：CL上で図面に赤入れ・コメント付与して協力会社に送れる
2. **修正前後の差分検出**：戻ってきた図面の修正箇所が瞬時に分かる
3. **指摘トラッキング**：出した指示が反映されているか追える

副要件：協力会社側UI（**登録不要・ゲストアクセス**）

### 18.4 ピン1集中原則との衝突 → 集中継続の判断

両建て（照査線・若手線並列）vs ピン1集中の衝突を再検討した結果、**集中継続**を選択：

| 理由 | 内容 |
|---|---|
| 若手線もWhole Product未成立 | 「現在の機能では価値が出ない」=2系統の未成立を同時走らせるのは死の谷加速 |
| 4ヶ月放置リスク | Part 13の「2025-12-15問題提起未実行」の再演リスク |
| ニッチ内浸透50%原則 | 別ピン分散だと福山1社の50%にも届かない |
| 「やめる3つ」原則 | 両建ては実質「コンセプト共感案件追跡」になる危険 |
