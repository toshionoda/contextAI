## 2. 解決すべき問題

| 観点 | 現状の問題 | 欲しい状態 |
|---|---|---|
| **無償（トライアル）** | データを聞かれる度に手動集計 | WAU・アクティブ率・稼働PJ数を自動出力 |
| **有償（チャーン）** | Churn Rateの定義・データソース未定義 | 月次解約アカウント数→Churn Rate自動計算 |
| **データ整合性** | ダッシュボードのスクリプト消失／経理と数字が合わない | Single Source of Truth確立 |
| **集計方法** | ポートエンジニア8件 vs AI6件のような差分 | 部門間で同じ定義・同じ数字 |

## 3. 設計の3つの論点（着手前に合意必要）

### 論点1：Churnの定義を統一する
- 解約数ベース? MRRベース? 「利用停止」も含む?
- アカウント単位 or 契約単位（戸田建設は1社で6名 → 1チャーン or 6チャーン?）
- カウント基準日（契約満了日 or 実停止日）

### 論点2：データソースの一元化方針
- Mixpanel（利用ログ）／Mazrica（契約・受注）／経理請求／有償化スプレッドシートが分散
- どれを"原本"にするか、または **dbt的な集約レイヤー** を作るか

### 論点3：配信先と更新頻度
- Slackパネル日次 / Notion週次 / 月次レポート
- 武井さん（営業ヘッド視点）／石丸さん（CFO的視点）／高取さん（PM視点）で見たい粒度が違う

## 4. 推奨アプローチ（3フェーズ）

### Phase 0（着手週）— 定義合意のための叩き台作成
- 武井／石丸／高取の3人が見たい指標を1枚にまとめる（インタビュー or 既存Slackスレ抽出）
- Churn定義の選択肢A/B/Cを比較表にして合意取り付け

### Phase 1（着手月内）— 最小ダッシュボード復活
- 既に存在する `civilinkkpi/` ディレクトリの資産活用
- Mixpanel + Mazrica + 有償化スプレッドシートを1本のスクリプトで毎朝集計
- Slackに `#kpi-daily` 投稿（既存の自動化パターン踏襲）

### Phase 2（翌月）— 指標体系の完成
- 無償WAU／稼働PJ数／Churn Rate／LTV／ARRを統合管理
- 経理側とも数字を突合する月次レポート
