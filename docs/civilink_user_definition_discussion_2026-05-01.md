## 2. 武井による前提整理（4月時点）

### 2.1 集計ロジック（建前 / 対外）

| 区分 | 集計ロジック | 根拠データ |
|---|---|---|
| **無料トライアルユーザー** | 「顧客管理（無料）」シートの phase = トライアル / トライアル準備中 の案件の組織メールアドレスを UserList から集計 | スプシ + UserList |
| **有償ユーザー** | 見積書 / 注文書 / 請求書のいずれかに記載の **契約開始日（=利用開始日 = 課金開始日）** を根拠とした、Mazrica の商品内訳入力データの日次集計 → **週次時点集計** | Mazrica |

### 2.2 武井の本音（社内整理）

無料トライアルユーザーの定義は2案ありえる：

1. ① **全ユーザー − 有償ユーザー** = 無料トライアル
2. ② **「顧客管理（無料）」シート集計** = 無料トライアル

→ ダッシュボード上は **②採用**

ただし武井自身が「**現状のアカウント数管理方法だと正確な無料アカウント数が把握できない**」と認めている。

### 2.3 必要な3区分

| # | 区分 | 性質 |
|---|---|---|
| 1 | 無料トライアル中のアカウント | アクティブな見込み客 |
| 2 | 無料トライアル終了後、有償移行しなかったアカウント | **死にアカウント** |
| 3 | 有償アカウント | 課金中 |

**現状、これら3区分を厳密に切り分けるロジックがない**。これが今回の議論の根本原因。

## 5. Churn Rate 議論

### 5.1 石丸の問い

> 当月中の解約アカウント数 ÷ 前月末有償アカウント数 を月次で計算したい。Mazrica生データ→AIで解約数が取れない。mixpanelで補足可能か？ MCPサーバ＋Claude連携も検討。

### 5.2 武井の回答

| 指標 | 算出方法 |
|---|---|
| チャーンアカウント数 | 集計表のアカウント数の **週次観測差分** |
| チャーン金額 | 集計表のMRR の **週次観測差分** |

**現実解**：「**前月末の前月で契約期間終了していて、当月更新がなかったアカウント数**」を集計

**freee vs 集計表**：
- freeeは月途中の契約・解約を考慮しきれない
- → 集計表の方が正確

### 5.3 「ユーザー数」と「契約数」の区別（武井）

> 同一人物が、基本機能とAIアドオンを2つ購入しているケースがあります。

| 集計単位 | 性質 |
|---|---|
| **契約数** | オーナー・エンジニア・サポート・AIアドオンを並列加算（同一人物が複数枠を持ちうる） |
| **ユーザー数** | 「誰が」を特定する必要があり、集計が困難 |

### 5.4 集計の基軸

| 視点 | 「ユーザー」の単位 | 用途 |
|---|---|---|
| 営業（武井） | 契約数（権限枠並列加算）／MRR | パイプライン管理 |
| 経理（村上・石丸） | 売り立ち＝請求書ベース、月按分 | 投資家報告・売上計上 |
| CS（野田・川端） | アクティブ実利用ユーザー（WaU/稼働PJ数） | 有償化CVR・チャーン予兆 |

→ **「ユーザー」と呼ぶ前にどの目線かを明示する**運用ルールがないと、毎週この議論が再発する。

## 7. 武井から開発部への依頼事項（提案ドラフト）

> 以下は武井さんの発言・整理ニーズから推定される、開発部に依頼したい機能要望のドラフトです。

### 7.1 アカウント区分の自動判定機能

**目的**: 「3区分（トライアル中／死にアカウント／有償）」を CS-Panel 上で自動判定できるようにする。

**要件**:
- UserList × 顧客管理（無料） × Mazrica の3ソースを結合
- 各ユーザーアカウントに以下の属性を自動付与
  - `account_status`: `trial_active` / `trial_expired_unpaid`（死にアカウント） / `paid` / `excluded`
  - `last_activity_date`: 最終イベント発生日
  - `paid_since`: 有償化日（あれば）
  - `paid_until`: 有償終了日（あれば）

### 7.2 「無料トライアルユーザー」定義の3層分離

**目的**: Dashboard 上で **権限定義 / 発行ベース / 稼働ベース** を別KPIとして並列表示する。

**要件**:
- Row 5 の単一値を以下に分割
  - `trial_users_authorized`（権限ベース）= 顧客管理（無料）T列合計
  - `trial_users_provisioned`（発行ベース）= UserListのトライアル組織ユニークユーザー数
  - `trial_users_active`（稼働ベース）= 直近◯日内にイベント発生したトライアル対象ユーザー数
- 3列の差分を可視化（差が出た時にアラート）

### 7.3 全ユーザー集計の除外ロジックの設定化

**目的**: 集計者によって異なる除外条件のばらつきを撲滅する。

**要件**:
- `Config.DASHBOARD_EXCLUDE_DOMAIN` をリスト化（`@malme.co.jp` 単独 → `[@malme.co.jp, @malme.gr.jp, ...]`）
- UserList に **「テストアカウント」フラグ列** を追加し、自動除外
- Dashboard に **「データ取得時刻」** を表示（毎日更新タイミングのズレを明示）

### 7.4 Churn 自動算出パイプライン

**目的**: 石丸さんが手動でChurn計算しなくて済むようにする。

**要件**:
- 武井集計表（週次差分ロジック）をスクリプト化し、毎週月曜に自動更新
- 出力：
  - チャーンアカウント数（週次差分）
  - チャーン金額（MRR週次差分）
  - 新規有償アカウント数（増分）
  - **ネット増減**（純増 − 解約）
- mixpanel 解約検知（最終イベントから◯日経過）と Mazrica 契約終了の **2系統で照合**

### 7.5 会社別の集計レイヤー追加

**目的**: 武井集計とMazrica/経理データの突合を可能にする。

**要件**:
- 集計表に **会社軸**（組織メールアドレス／会社名）の列を追加
- 同一会社内の **権限別アカウント数** を会社単位で展開
- 経理側の請求書ベース集計と **会社×月** で1対1突合できる形式

### 7.6 GAS / Dashboard スクリプトのバージョン管理

**目的**: 「スクリプトが消えた」事故の再発防止。

**要件**:
- CS-Panel関連のスクリプトを **Gitリポジトリに集約**（既に `mixpanel/cs_panel/` は存在）
- GAS側は **clasp** で同期、main保護
- 編集前に必ずローカル取得 → コミット → push の運用化
- DLしたExcelに頼らず、コードレビューを通る運用に切り替え

### 7.7 アカウント状態の時系列保存

**目的**: 「416の時系列データが整理できていない」状態の解消。

**要件**:
- 毎日のスナップショット（全ユーザー数 / 有償 / トライアル / 稼働）を時系列テーブルに保存
- **Dashboard上の各KPIが過去任意時点の値で参照可能**
- 集計ルールが変わった場合、ルール変更日を明示し、変更前後で再計算しない（不変性の担保）
